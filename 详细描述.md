## 二、总体架构图与功能说明

> "1+N+1+N"（PFAS）架构体系围绕基层治理"数据怎么管、能力怎么扩、智慧怎么用、问题怎么解"四大核心问题，构建了四层协同的数字化治理闭环：

| 层级 | 定位 | 能做什么 |
|------|------|---------|
| **第一层：基础平台（Platform）** | 统一底座 | 以"一标三实"数据中台为核心，将人、房、事、物、组织五要素以标准地址为锚点统一建模，实现**一次采集、多方复用**；叠加GIS时空底座、周边智能推送、格友圈协同，为上层提供数据、空间、协同三大基础服务 |
| **第二层：生态平台（Federation）** | 能力插件 | 横向联接 **7类智能硬件**（AI眼镜、无人机、二维码门牌、融合通讯、IoT传感器、AI摄像头、北斗终端）和 **4类政务软件**（赣政通、E呼即办、智慧调度、效能评估），每个模块即插即用、可独立交付，数据统一汇聚至基础平台 |
| **第三层：AI赋能中心（AI）** | 智慧大脑 | 基于"三层AI体系"（能力底座→专题Agent工厂→AI运营中心），提供 **5个专题Agent**——采集Agent（口述即录入）、巡查Agent（AR+AI隐患识别）、调解Agent（随身法律顾问）、分拨Agent（秒级精准派单）、预判Agent（治理天气预报），让基层"少动手、少跑腿、少犯错" |
| **第四层：治理场景（Scenarios）** | 业务呈现 | 面向公安、住建、综治、街道等不同角色，按需组合下层能力，快速搭建**靶向解决一线痛点**的复合应用——立体巡防、扫码报修、一码共治、态势一张图等，新场景通过"五大要素判定法"快速匹配复用 |

**一句话总结：** 用一个厚底座沉淀共性能力、用N个开放插件联接内外生态、用一个智慧大脑贯穿智能、用N个敏捷场景靶向服务——让基层干部**少录一次、少跑一趟、少写一份、少等一天**。

## 三、第一个"1"：基础平台（Platform）—— 共性能力中台

### 3.1 定位

基础平台是整个"1+N+1+N"体系的**统一底座**，由公司自主研发，承担"能力沉淀、数据汇聚、标准统一、服务开放"四项核心职责。它是"厚底座"战略的物理载体——所有上层能力（生态平台、AI赋能中心、治理场景）必须通过底座才能访问数据、调用服务、触达用户。

**一句话：** 不让每个场景从零开始，把四大共性能力一次性做扎实。

### 3.2 四大共性能力

#### 3.2.1 数据中台与治理数据湖

这是底座最核心的能力，也是"一标三实"数据模型的承载层。

**一标三实数据模型：** 以标准地址（一标，GB/T 35639-2017）为锚点，串联实有人口、实有房屋、实有单位（三实），形成"人—房—事—物—组织"五大要素的标准化数据模型。简单说，二维码门牌就是每个地址的"数字身份证"，扫码就知道这个地方住了谁、有什么单位、有什么风险。

| 能力 | 说明 |
|------|------|
| 多源数据汇聚 | 汇聚公安、综治、住建、城管、民政等多部门数据，形成辖区全域"块数据" |
| 数据治理与清洗 | 地址标准化（GB/T 35639-2017）、身份证号校验、重复数据合并、失效数据标记，从源头保证数据质量 |
| 数据鲜活度维护 | 通过AI眼镜巡查采集、网格员走访更新、群众自主申报、无人机航拍比对等多渠道持续更新，底数准确率保持99%以上 |
| 数据资产沉淀 | 在治理运行中自然累积两大数据资产：部门处置的"运行数据"（工单流转、处置时效、满意度），和地址下的"块数据"（人口、房屋、事件），为远期数据产品化奠定基础 |

**解决了什么问题：** 过去公安、综治、住建各建系统，同一栋楼在不同系统里地址写法都不一样，同一户人口要在三个系统里重复录入。现在以国标地址编码为唯一锚点，实现**一次采集、多方复用**，从源头终结数据孤岛。

#### 3.2.2 时空地理信息GIS底座

| 能力 | 说明 |
|------|------|
| 统一地图服务 | 二维/三维地图渲染、卫星影像切换、POI搜索、路径导航。政务内网侧用天地图，互联网便民侧用高德/百度地图SDK |

## 四、第一个"N"：生态平台（Federation）—— 能力模块集合

## 一、第一个"N"的定位与核心定义

### 1.1 一句话定义

**第一个"N"是一组可插拔、可扩展的能力模块集合。** 每个模块既可以作为独立产品对外交付，也可以融入基层智治综合协同平台（第一个"1"），与其他模块组合形成完整的治理解决方案。

### 1.2 核心特征

| 特征 | 说明 |
|------|------|
| **即插即用** | 每个模块通过标准化API与基础平台对接，不依赖特定品牌或供应商 |
| **可独立交付** | 任一模块可拆分出来单独销售和部署，如单独交付"无人机自动巡飞系统" |
| **可融合协同** | 模块间天然具备协同关系——无人机发现问题→AI眼镜地面核查→事件中心闭环处置 |
| **持续扩展** | N不是固定数字，随业务发展持续增加，新增模块遵循统一接入规范即可入列 |

### 1.3 与第一个"1"（基础平台）的关系

```
第一个N（生态平台）  ←→  第一个1（基础平台）
    "插件层"              "底座层"

融合使用：N的能力通过1的数据底座、GIS、融合通讯、事件中心等共性服务，
          向治理场景层输出完整的解决方案。

独立使用：N的模块可脱离1单独运行——例如无人机系统可独立部署，
          通过标准接口与客户现有平台对接。
```

**核心理念：** 联接而非替代。市场上已有的优秀硬件和政务系统，不做重复建设，通过标准化接口实现"即插即用"。
**数据主权原则：** 所有通过本平台各模块采集的数据，统一汇聚至基层平台数据中台（一标三实数据模型），平台是数据的**唯一汇聚点和统一出口**。各归口部门（公安、消防、城管、民政等）**按需从本平台同步**所需数据至各自业务系统，无需重复采集。平台是数据的统一汇聚点和分发枢纽，各部门按需获取数据服务。这一原则确保：①基层只录一次，多方按需复用；②数据鲜活度由平台统一维护，各部门拿到的永远是"最新版本"；③数据安全和隐私保护在平台层统一管控，而非分散在各部门系统。

## 二、分类方式

### 2.1 主分类：按产品形态（推荐）

经过对现有模块的梳理和扩展，第一个N采用**"智能硬件类 + 本地政务软件类"**两级分类。这个分类方式直观、易懂，且与采购决策和部署模式直接对应：

| 分类 | 产品形态 | 部署特点 | 采购方式 |
|------|---------|---------|---------|
| **智能硬件类** | 物理设备 + 配套软件 | 有物理交付，需现场安装调试 | 硬件采购 + 软件服务 |
| **本地政务软件类** | 纯软件/平台对接 | 接口对接为主，少量定制开发 | 软件服务/数据接口 |

### 2.2 数据主权与流向原则（重要）

所有N模块产生的数据，遵循统一的数据主权规则：

```
采集层（N模块：AI眼镜/无人机/二维码门牌/IoT传感器/摄像头/北斗终端）
    ↓ 全部数据统一汇入
存储层（基层平台数据中台——"一标三实"唯一数据源）
    ↓ 按需同步至各部门系统
使用层（公安人口库 / 消防监管系统 / 城管执法系统 / 民政养老系统 / ...）
```

| 原则 | 说明 |
|------|------|
| **平台是唯一数据源** | 所有采集数据统一存储在基层平台数据中台，不做分散存储 |
| **部门按需同步** | 部门从平台同步所需数据，或者按需平台推送原始数据 |
| **基层只录一次** | 网格员/民警/检查员在平台录入一次，多部门按权限复用，杜绝重复采集 |
| **同步方向为"平台→部门"** | 数据流向始终是"基层平台→部门系统"，部门系统不直接向平台写入数据（部门审核确认通过API接口回传至平台） |

## 三、智能硬件类 —— 各模块详细介绍

### 3.1 AI眼镜

#### 功能描述

AI眼镜是专为基层一线工作人员打造的**解放双手的智能巡查终端**。智能AI眼镜本身即为一体化电子眼镜设备——摄像头、骨传导耳机、显示模块全部集成在眼镜上，所有信息内容直接在眼镜镜片上呈现，无需外接抬头显示器（HUD）。工作人员戴上眼镜即可实现语音交互、AI视觉识别、远程协作三大核心能力，真正解放双手。

**主管部门与政策依据：**

| 维度 | 说明 |
|------|------|
| **牵头部门** | 上饶市大数据局已牵头完成全市二维码门牌统一编码和现场贴牌，AI眼镜扫码即进入本平台，入口基础已具备 |
| **使用主体与归口** | ①网格员/综治人员→政法委（综治中心）管理 ②民警/辅警→公安局管理 ③消防检查人员→消防救援支队管理 ④城管→城市管理局管理 |
| **采购路径** | 网格员装备→政法委综治经费/街道工作装备经费；民警装备→公安警务装备经费；消防检查装备→消防救援支队业务经费。建议市大数据局统一招标、按部门分签合同以降低单价 |
| **政策依据** | 中央政法委市域社会治理现代化试点要求"运用智能化手段提升治理效能"；《"十四五"国家信息化规划》推进基层智慧治理 |

**核心能力清单：**

| 能力 | 说明 |
|------|------|
| 语音上报 | 口述事件内容，AI自动转写为结构化工单 |
| AI视觉识别 | 实时识别可疑人员、车辆、隐患（电线私拉、通道堵塞、灭火器过期等） |
| OCR证件识别 | 扫一眼身份证/营业执照，自动提取信息填入表单 |
| AR导航引导 | HUD投射检查清单，逐项引导，防止漏项 |
| 远程标注/直播 | 第一视角画面实时回传指挥中心，后方专家远程标注指导 |
| 免提扫码 | 看向二维码门牌自动识别（市大数据局已完成全市门牌编码和贴牌），无需掏手机 |
| 离线工作 | 无网络环境下端侧AI继续工作，回网自动同步 |

#### 三个最常用场景（结合基层平台）

**场景一：入户信息采集 —— "看一眼就填好"**

> 网格员小张接到走访独居老人的任务。他戴着AI眼镜走进小区，看向门牌二维码自动识别地址（市大数据局已统一编码贴牌），平台自动调出该户历史档案。他口头询问老人近况，眼镜自动语音转写成走访记录。老人出示身份证，眼镜OCR瞬间填入姓名、身份证号。过去15分钟的采集流程，现在3分钟完成。

- **协同模块：** 二维码门牌（大数据局统一编码，扫码定位）、数据中台（调用历史档案）、赣政通（接收走访任务）
- **增效数据：** 采集效率提升80%（15分钟→3分钟）
- **数据流向：** AI眼镜采集的全部数据→基层平台数据中台统一存储（唯一数据源）→按政法委综治中心设定的权限规则，按需同步至公安人口库、民政关爱对象库等相关部门系统

**场景二：安全巡检 —— "AR带着走，一项不漏"**

> 消防检查员老李到商铺做季度消防检查。眼镜HUD逐项显示检查清单，AI自动识别灭火器有效期和疏散通道状态。发现通道堆有纸箱，自动拍照+标记+生成整改通知书。40分钟检查压缩到15分钟，漏项率从15%降到2%以下。

- **协同模块：** 事件中心（自动生成整改工单）、低代码表单引擎（检查清单模板）、电子地图（位置标记）
- **增效数据：** 巡检效率提升62.5%，漏项率降低87%
- **数据流向：** 消防检查记录→基层平台数据中台→按需同步至消防救援支队消防监管系统；整改工单→平台事件中心生成→按规则同步至城管/市监等部门系统

**场景三：纠纷现场联合处置 —— "前后方一起看、一起判"**

> 两户居民因房屋边界问题产生纠纷，网格员小王到场后打开AI眼镜直播。指挥中心（政法委综治中心）的调解专家通过融合通讯系统看到第一视角画面，远程标注关键位置。同时打开电子地图比对历史测绘数据（自然资源局提供的基础地理数据）。前后方协同，10分钟完成现场勘查，当场给出调解方案。

- **协同模块：** 融合通讯调度引擎（直播+对讲）、电子地图（空间坐标比对，底图来源：自然资源局天地图）、事件中心（记录处置过程）
- **数据流向：** 调解全程记录→基层平台数据中台存储→如需司法确认，平台按授权将调解文书同步至司法局人民调解系统

### 3.2 无人机自动巡飞系统

#### 功能描述

### 3.5 物联网传感器网络（新增模块）

#### 功能描述

物联网传感器网络是**城市运行状态的"神经末梢"**，通过在城市关键节点部署各类智能传感器，实现对环境、安全、设施的24小时自动感知和异常预警。

**关键设计原则：** 传感器设备由**各归口部门按职责分别采购、分别部署、分别维护**，平台提供统一数据接入标准（NB-IoT/Cat.1）和告警→工单→分拨的共性能力。不搞"大一统采购"，尊重政府各部门的职责边界。

**按归口部门分列的传感器清单：**

##### 一、消防救援支队归口——火灾防控类

| 设备类型 | 监测内容 | 部署位置 | 采购路径 | 政策依据 |
|---------|---------|---------|---------|---------|
| 智能烟感 | 烟雾浓度、温度异常 | 老旧小区、出租屋、"九小场所" | 消防救援支队火灾防控专项经费/街道消防安全经费 | 《消防法》第九条、国务院《消防安全责任制实施办法》 |
| 电气火灾监测 | 电流异常、线路温度、剩余电流 | 老旧楼栋、电动车充电棚 | 消防救援支队+住建局联合部署 | 国务院安委会相关工作方案 |

> **告警流转：** 传感器告警→平台事件中心→自动生成消防类工单→推送消防救援支队+属地街道+责任网格员

##### 二、水利局归口——防汛抗旱类

| 设备类型 | 监测内容 | 部署位置 | 采购路径 | 政策依据 |
|---------|---------|---------|---------|---------|
| 水位监测仪 | 河道/管网水位、流速 | 易涝点、河道闸站、水库 | 水利局防汛抗旱专项经费 | 《防洪法》、国家防总防汛抗旱信息化建设要求 |

> **告警流转：** 水位异常→平台事件中心→推送水利局+应急管理局+属地街道
> **城管局协作：** 城市下穿隧道、地下车库等市政设施的积水监测由城管局（市政管理处）采购水位传感器，告警同步推送城管+交警

##### 三、城管局归口——市政设施类

| 设备类型 | 监测内容 | 部署位置 | 采购路径 | 政策依据 |
|---------|---------|---------|---------|---------|
| 窨井盖传感器 | 井盖位移/倾斜/打开、井下水位 | 主干道、学校周边、商业区 | 城管局市政设施管理经费，各权属单位（供水/供电/通信/燃气）分别承担本行业井盖 | 住建部《关于进一步加强城市窨井盖安全管理的通知》 |

> **告警流转：** 井盖异常→平台事件中心→自动识别权属单位→推送至对应权属单位抢修班组。权属数据由城管局在部署时统一录入平台

##### 四、自然资源局归口——地质灾害类

| 设备类型 | 监测内容 | 部署位置 | 采购路径 | 政策依据 |
|---------|---------|---------|---------|---------|
| 地灾监测 | 边坡位移、土壤含水率、裂缝变化 | 地质灾害隐患点（省级/市级台账） | 自然资源局地质灾害防治专项经费 | 《地质灾害防治条例》、自然资源部监测预警体系建设要求 |

> **告警流转：** 达到预警阈值→平台自动通过融合通讯外呼乡镇负责人和隐患点巡查员→同时推送自然资源局+应急管理局+属地乡镇

##### 五、生态环境局归口——环境保护类

| 设备类型 | 监测内容 | 部署位置 | 采购路径 | 政策依据 |
|---------|---------|---------|---------|---------|
| 扬尘噪音监测 | PM2.5/PM10、噪音分贝 | 工地、主干道、工业区 | 生态环境局环境监测经费；住建局配合工地端部署 | 《大气污染防治法》、《环境噪声污染防治法》 |

> **告警流转：** 扬尘/噪音超标→平台事件中心→生成环保类工单→推送生态环境局+住建局（如为工地源头）

##### 六、平台提供的共性能力

| 能力 | 说明 |
|------|------|
| 统一数据接入标准 | 指定NB-IoT/Cat.1为平台兼容的传感器通信协议，各归口部门采购设备须符合此标准 |
| 统一告警→工单转换 | 所有传感器告警统一转换为事件中心标准化工单，按归口部门自动匹配推送目标 |
| 统一GIS展示 | 所有传感器数据在电子地图上分图层展示，各部门可叠加查看自己归口的传感器图层 |
| 统一统计分析 | 大数据分析平台汇总各部门传感器运行数据，生成跨部门综合分析报告 |

#### 三个最常用场景（结合基层平台）

**场景一：老旧小区火灾综合防控 —— "烟感+电气监测+平台联动"**

> 某老旧小区（住建局旧改计划内）同时部署了消防支队的智能烟感和消防+住建联合的电气火灾监测。凌晨2点，烟感报警（消防归口）且电气监测也检测到电流异常。平台同时收到两个告警，自动生成消防类紧急工单：①匹配地址→XX小区3栋502室+该户居住3人（含1名老人，公安/民政数据）；②融合通讯自动拨打房东+最近网格员+微型消防站；③GIS地图标注告警位置。网格员5分钟到场断电灭火。

- **协同模块：** 数据中台（地址匹配+住户信息，公安/民政数据）、融合通讯（自动外呼）、电子地图（告警定位）、事件中心（消防类工单流转）
- **核心价值：** 火灾发现从"起火后"提前到"冒烟时"
- **数据流向：** 烟感+电气监测告警→平台事件中心→自动生成工单+匹配住户信息（平台自有数据）→推送消防救援支队火警系统+属地网格员。**告警数据在平台留存**，消防支队按需同步至其火灾防控系统

**场景二：城市内涝"水利+城管"联合防控 —— "水位一涨，两部门同时响应"**

> 暴雨红色预警期间，水利局32个河道水位监测点和城管局18个下穿隧道积水监测点实时回传数据。某隧道水位5分钟从0.3米飙升至0.8米——城管局积水传感器触发告警，水利局上游500米水位监测也显示异常升高。平台综合分析判定为"隧道内涝风险"，同时触发：①标记该路段"禁止通行"同步导航地图；②推送城管市政抢修班组+交警；③推送水利局开闸排水调度。三个部门在平台上看到同一套数据、同一个研判结论。

- **协同模块：** 电子地图（多部门传感器叠加展示+分色预警）、融合通讯（自动外呼+群呼）、事件中心（多部门并发推送）
- **核心价值：** 打破水利/城管/交警三部门信息壁垒
- **数据流向：** 水利+城管传感器数据→实时汇聚至平台数据中台→平台综合分析判定风险→生成工单推送水利局/城管局/交警。**各部门传感器数据汇聚至平台形成全域视图**

**场景三：窨井盖"权属自动识别" —— "井盖一动，平台自动判断该谁去修"**

> 某小学门口窨井盖倾斜传感器触发告警。平台自动读取该井盖的"权属标签"（城管局部署时录入：XX路15号，权属=市自来水公司，联系人=张师傅/139XXXX）。自动生成工单→推送自来水公司抢修班组+城管局市政管理处（监督方）。抢修人员15分钟到场更换，拍照闭环。

- **协同模块：** 电子地图（精确坐标+权属标注）、事件中心（按权属自动分拨→城管局监督）、赣政通（任务推送至权属单位抢修班组）
- **核心价值：** 权属不清导致推诿扯皮的问题被"权属标签"机制解决
- **数据流向：** 井盖传感器告警→平台事件中心→平台根据城管局维护的权属标签自动匹配→推送至对应单位抢修系统。**权属标签由城管局维护并同步至平台，告警数据在平台留存**

### 3.6 AI智能监控摄像头网络（新增模块）

#### 功能描述

AI智能监控摄像头网络是**城市固定点位的"不眨眼的哨兵"**，与无人机（移动巡飞）形成"固定哨+巡逻兵"的立体感知体系。依托政法委"雪亮工程"已建成的摄像头基础设施，叠加AI视觉分析能力，在不新建硬件的前提下，让现有摄像头从"被动录像"升级为"主动发现"。平台作为技术支撑方，在公安、民政、城管等多部门授权框架下，为基层治理提供实时事件识别和民生服务能力。

**政策依据：** 中央政法委"雪亮工程"建设要求推动视频监控智能化升级；九部委《关于加强公共安全视频监控建设联网应用的若干意见》。

**与无人机的互补关系：**

| 维度 | 无人机（巡逻兵） | AI摄像头（固定哨） |
|------|--------|------------|
| 覆盖方式 | 移动巡飞，周期覆盖 | 固定点位，7×24小时持续 |
| 适用场景 | 大面积巡查、高空视角、灾后评估 | 重点部位持续监测、夜间安防、人流态势 |
| 发现时效 | 按航线周期（小时级） | 实时（秒级） |
| 归属条件 | 平台统一管理 | 政法委（雪亮工程）建设，平台做AI分析叠加 |
| 协同关系 | 摄像头发现异常→触发无人机起飞侦察 | 无人机发现目标→联动最近摄像头持续跟踪 |

**核心能力清单：**

| 能力 | 说明 |
|------|------|
| AI实时识别 | 人群异常聚集、烟火检测、重点部位入侵、交通拥堵、老人/儿童异常徘徊（面部模糊处理） |
| 事件自动生成 | AI识别异常→截图+时间戳+坐标→自动生成事件工单→流入事件中心 |
| 多摄像头联动 | 走失人员跨区域接力查找、可疑行为跨点位追踪 |
| 无人机联动 | 摄像头发现异常→自动触发最近无人机起飞，空地协同确认 |
| 复用现有基础设施 | 依托"雪亮工程"已部署摄像头，叠加AI分析，边际成本低 |
| 隐私合规（关键） | 人脸/车牌自动模糊处理，录像分级授权查看，操作日志全审计。《个人信息保护法》合规设计——公共场所AI分析仅做行为识别（"有人群聚集"），不做个体身份持续追踪 |

#### 三个最常用场景（结合基层平台）

**场景一：走失人员快速查找 —— "奶奶走丢了，摄像头+无人机+网格员一起找"**

> 傍晚6点，一位居民焦急地到社区求助——他78岁的老母亲下午出门散步后一直未归。网格员立即在平台上发起"走失人员查找"流程：①录入老人的体貌特征和最后出现的地点；②平台调取该地点周边雪亮工程摄像头最近2小时的画面，AI自动筛选出匹配特征的行人轨迹——发现老人曾在3个点位出现，最后方向是往城郊公园；③系统自动触发无人机起飞前往公园区域高空搜索；④同时将老人照片和最后位置推送给公园周边的3名网格员协助查找。25分钟后，无人机热成像在公园深处发现一名独坐的老人，网格员赶到确认正是走失老人，平安带回。从求助到找到，全程38分钟。

- **协同模块：** 雪亮工程摄像头（画面调取+AI检索）、无人机（自动起飞+热成像搜索）、电子地图（轨迹标注+网格员位置）、事件中心（走失求助流程）、赣政通（推送给周边网格员）
- **核心价值：** 走失人员找回时间从平均4小时缩短至30分钟。**这是一个"服务"场景，不是"管控"场景**

**场景二：大型活动人流安全引导 —— "不是监控你，是帮你避开拥堵"**

> 元宵灯会期间，主城区核心路段人流量激增。部署在关键路口的AI摄像头实时统计人流密度（仅计人数，不采集个体身份），在GIS地图上以热力图呈现。当某桥段人流密度超过安全阈值时，平台自动触发三级措施：①通过融合通讯推送提醒给现场执勤的安保人员（配AI眼镜）前往疏导；②联动周边3块市政LED大屏发布实时引导——"XX桥当前人流密集，建议从YY桥绕行"；③在赣服通"便民服务"中推送实时路况，引导市民避开拥堵区域。活动期间未发生一起踩踏或走失事件。

- **协同模块：** 电子地图（人流热力图+阈值预警）、融合通讯（推送执勤人员）、AI眼镜（现场疏导第一视角）、赣服通（便民路况推送）
- **核心价值：** 从"出了事故再处置"到"事前引导保安全"，保障群众安全的同时保护了隐私

**场景三：校园周边安全守护 —— "不是盯着谁看，而是守护该守护的人"**

> 下午放学时段，某小学门口的AI摄像头检测到一名男子长时间在校门口停留（超过15分钟），系统判定为"异常徘徊"（AI仅识别行为模式，不采集面部身份）。平台自动执行：①生成安全预警工单→推送至该校"护学岗"责任民警的赣政通；②民警收到提醒，3分钟到场盘查，确认该男子为精神障碍患者（曾在该校周边骚扰学生），妥善劝离；③事件记录存入平台，作为该校周边安全态势的数据积累。全程AI只做了"行为识别"，未采集任何个体身份信息。身份确认是民警到场后按正常执法程序完成的。

- **协同模块：** 事件中心（自动生成预警工单）、电子地图（坐标定位+护学岗匹配）、融合通讯（自动提醒）、赣政通（任务推送）
- **核心价值：** 用AI"行为识别"替代"人脸识别"，在保护隐私的前提下守护校园安全

### 3.7 北斗定位关爱终端（新增模块）

#### 功能描述

北斗定位关爱终端是**面向民政养老服务体系的特殊群体安全守护设备**，以智能胸卡/手环形态，为独居老人、失智老人等政府重点关爱群体提供实时定位、电子围栏、SOS求助、异常预警四大核心能力。终端告警与平台事件中心+网格体系联动，形成"智能发现→自动派单→网格员上门→闭环反馈"的完整关爱链条。这是基层智治平台落实**"老有所安"**的具体抓手，也是民政部门养老服务数字化转型的切入点。

**政策依据：** 国务院《"十四五"国家老龄事业发展和养老服务体系规划》要求"推进智能化养老服务"；民政部"十四五"规划明确"建立特殊困难老年人探访关爱服务机制"；江西省政府购买养老服务政策为终端采购提供资金渠道。

**主管部门与采购路径：**

| 维度 | 说明 |
|------|------|
| **主管部门** | 民政局养老服务科，平台作为技术支撑和数据服务方 |
| **资金渠道** | 民政养老服务专项资金 / 政府购买养老服务项目 |
| **发放对象** | 独居老人、失智老人、高龄空巢老人等纳入民政关爱名单的特殊困难群体，政府免费发放 |
| **数据管理** | 轨迹数据加密存储，仅授权家属+责任网格员可见，民政部门可查看辖区整体态势（脱敏统计） |

#### 三个最常用场景（结合基层平台）

**场景一：独居老人"寂静预警" —— "老人24小时没出门，系统替你去敲门"**

- **协同模块：** 事件中心（自动生成关爱任务）、智慧调度（匹配责任网格员）、赣政通（任务推送）、数据中台（特殊群体档案）、电子地图（网格员导航）、大数据分析（异常轨迹检测）
- **核心价值：** 独居老人意外发现时间从"天级"缩短至"小时级"，让"老人在家出事无人知"的悲剧不再发生
- **主管部门：** 民政局养老服务科

**场景二：失智老人"安心围栏" —— "爷爷走出小区，家人和网格员同时收到提醒"**

- **协同模块：** 电子地图（电子围栏+实时位置+家属导航）、事件中心（告警分发+计时升级）、赣政通（网格员任务推送）、融合通讯（一键联系家属）
- **核心价值：** 失智老人走失找回时间从平均4小时缩短至15分钟
- **主管部门：** 民政局养老服务科（设备发放）、政法委综治中心（网格员联动）

**场景三（扩展场景）：社区矫正人员轨迹监管 —— "在司法局授权和监管规则下运行"**

> *此场景为扩展应用，归属司法局社区矫正科主管，需单独授权开通。数据、权限、流程与民政养老场景完全隔离。*

### 4.1 赣政通（任务推送与协同办公）

#### 功能描述

赣政通是江西省统一政务协同办公平台，覆盖省—市—县—乡四级、联通超万个政务部门。本平台与赣政通深度对接，实现身份互信、待办同步、任务推送、督查联动四大核心能力，让基层人员在赣政通一个入口处理所有待办。

**核心对接能力：**

| 对接项 | 说明 |
|--------|------|
| 身份互信 | 通过省统一身份认证平台实现互信登录，一个账号通两平台 |
| 待办同步 | 本平台任务自动推送至赣政通待办中心，签收/完成状态双向同步 |
| 消息通知 | 本平台预警、提醒通过赣政通消息通道推送 |
| 督查对接 | 超时未处置事件可同步至赣政通督查督办模块，触发逐级催办 |

#### 三个最常用场景（结合基层平台）

**场景一：巡查任务"自动推、自动收" —— "不用盯屏幕，来了就干"**

> 网格员老陈早上8:30打开赣政通，待办中心列出今天3项任务：①9:00前巡查XX路沿街商铺消防安全（系统识别上次检查已超过1个月）；②走访3户独居老人（按周期自动生成）；③核实昨天无人机发现的XX小区楼顶疑似违建。老陈逐一点击签收，出发执行。每完成一项提交结果，赣政通待办自动消除，本平台同步记录完成状态。全程无需在多个系统间切换。

- **协同模块：** 本平台事件中心（任务生成）、无人机系统（违建线索）、数据中台（走访周期触发）
- **核心价值：** 任务指派从"微信群@人"升级为"系统自动推送+跟踪"
- **数据流向：** 任务生成→本平台事件中心→推送至赣政通待办中心→签收/完成状态回传本平台。**平台是任务数据的唯一记录系统**，赣政通是消息推送和签收前端

**场景二：超时事件"逐级催办" —— "到点了没办，系统不答应"**

> 某小区垃圾堆积投诉工单已超处置时限4小时，网格员未签收。系统自动执行三级催办：①推送赣政通消息提醒网格员；②1小时后仍未响应→推送至其主管；③再1小时后→推送至街道分管领导。同时，工单数据同步至赣政通督查督办模块。机制保障下，平台工单及时签收率达到98%以上。

- **协同模块：** 本平台事件中心（超时检测）、赣政通（督查督办）
- **核心价值：** 用"制度+系统"替代"人盯人"，事件超时率下降80%
- **数据流向：** 超时检测→本平台事件中心→催办指令同步至赣政通督查督办模块→催办结果回传本平台记录。**催办规则和超时判断在平台侧执行**

**场景三：跨部门"公文+事件"联动 —— "一个事件触发，多个部门齐动"**

> 某河道出现大量漂浮物，网格员上报后事件自动流转至水利局。水利局处置后结案时，系统提示："该河段近3个月已出现4次同类事件，建议修订河道保洁制度。"水利局经办人在赣政通中直接起草《关于加强XX河道保洁工作的通知》，利用赣政通公文流转功能——签收→拟办→批办→下发至沿线3个乡镇。从事件发现到制度修订、文件下发，全程在同一平台完成。

- **协同模块：** 本平台事件中心（事件研判）、AI赋能中心（高频事件识别）、赣政通（公文流转）
- **核心价值：** 打通"事件处置→问题研判→制度修订"的完整闭环
- **数据流向：** 事件研判结果→本平台事件中心→水利局经办人在赣政通起草公文（使用赣政通公文流转功能）→公文下发结果回传本平台关联原事件。**事件数据在平台，公文流转在赣政通**

### 4.2 E呼即办（民生诉求联动）

E呼即办是上饶市创新建设的民生诉求快速响应平台，整合110、12345热线、社会治理网格三张网，18家高频部门常驻联合指挥中心。以"12530"机制（1分钟成单→2分钟派单→5分钟签收→30分钟到场）实现极速响应。本平台作为后台数据底座和协同处置工具，与E呼即办形成"前台指挥+后台支撑"的双平台联动。

**核心对接能力：**

| 对接项 | 说明 |
|--------|------|
| 工单双向同步 | E呼即办工单同步至本平台；网格员上报事件推送至E呼即办 |
| 标准地址匹配 | E呼即办工单通过本平台"一标三实"底座自动解析地址 |
| 处置过程互通 | 网格员处置记录实时回传E呼即办 |
| 融合通讯联动 | E呼即办调取本平台无人机图传/AI眼镜直播流 |
| 吹哨对接 | 网格员一键发起E呼即办"吹哨报到"流程 |

#### 三个最常用场景（结合基层平台）

**场景一："12530"极速响应 —— "1分钟成单、30分钟到场"**

- **协同模块：** 数据中台（地址解析+网格匹配）、融合通讯（现场图传）
- **核心价值：** 平均办结时长从44小时压缩至12小时（缩短73%）
- **数据流向：** 市民诉求→E呼即办生成工单→同步至本平台事件中心→本平台地址解析+网格匹配→责任单位确认结果回传E呼即办。**地址解析和网格匹配由本平台提供**，E呼即办是前台指挥

- **协同模块：** 事件中心（吹哨记录+部门评分）、赣政通（任务推送）、融合通讯（联合调度）
- **核心价值：** 已累计处理吹哨事件92件，延伸解决问题370余个，满意率96.5%
- **数据流向：** 吹哨记录→本平台事件中心→同步至E呼即办指挥中心审核→部门报到指令下发→处置结果回传本平台+E呼即办。**吹哨全流程数据在平台留存**

**场景三："未呼先办"主动治理 —— "不等群众投诉，问题已经解决"**

- **协同模块：** AI赋能中心（数据洞察与预判）、数据中台（历史事件分析）、事件中心（工单趋势）
- **核心价值：** 从"接诉即办"升级为"未呼先办"

### 4.3 智慧调度指挥系统（新增模块）

#### 功能描述

智慧调度指挥系统是**基于GIS和AI的事件智能分拨与资源优化调度平台**。通过"事件自动分类→地址精准匹配→规则引擎分拨→资源最优配置"四级调度链，实现从"人工判断派给谁"到"系统辅助算出最优解"的升级。

**调度授权来源（关键）：**

> 跨部门调度不是技术问题，是组织授权问题。本系统的调度指令必须建立在明确的组织授权基础之上。

| 调度场景 | 授权主体 | 授权依据 | 调度范围 |
|---------|---------|---------|---------|
| **网格员日常调度** | 政法委（综治中心） | 中央《关于加强基层治理体系和能力现代化建设的意见》明确综治中心承担"指挥调度"职能 | 网格员、综治工作人员 |
| **应急指挥调度** | 应急管理局 | 《突发事件应对法》授予应急管理局法定应急指挥权。应急状态下调度范围可扩展至所有参与响应的部门和人员 | 所有参与应急响应的部门 |
| **民生诉求联动调度** | E呼即办联合指挥中心 | 上饶市E呼即办18家部门常驻联合指挥机制，"12530"流程已制度化授权 | E呼即办常驻的18家部门 |
| **部门内部调度** | 各归口部门自行管理 | 各部门对本部门人员有天然管理权 | 各部门内部人员 |

**核心设计原则：**

1. **分权而非集权** —— 不同场景的调度指令来自不同的法定授权主体，系统不越权调度
2. **规则前置** —— 调度规则（谁可调度谁、什么条件下可调度）由各归口部门事先确认并录入系统
3. **应急兜底** —— 应急状态下（应急管理局确认"进入应急模式"），调度权限临时升级、范围临时扩大，应急结束自动恢复

**核心能力清单：**

| 能力 | 说明 |
|------|------|
| 事件智能分类 | NLU自动识别事件类型、紧急程度、涉及部门，分类规则由政法委综治中心统一维护 |
| 地址精准匹配 | 基于"一标三实"自动解析地址，匹配责任网格和责任部门 |
| 规则引擎分拨 | 日常模式（政法委综治中心规则）和应急模式（应急管理局规则），规则修改须经授权主体审批 |
| 资源最优配置 | 综合距离、技能、忙闲状态推荐最佳处置人员。推荐结果供指挥员确认后执行（非自动下达） |
| 多部门协同调度 | 日常→综治中心协调；应急→应急管理局授令。全过程记录可追溯 |

#### 三个最常用场景（结合基层平台）

**场景一：日常事件"智能分拨" —— "算法推荐+人工确认，政法委综治中心授权"**

> 市民投诉"XX路烧烤店油烟扰民"。系统在政法委综治中心设定的日常调度规则下运行：NLU识别为"餐饮油烟"（城管主责+环保协办）→地址匹配至"城南街道第3网格"→规则匹配"油烟→城管+环保，2小时到场"（城管局和环保局事先确认的规则）→推荐城管老王为主办人。推荐结果推送综治中心值班员确认后下达。

- **协同模块：** 数据中台（地址匹配）、AI赋能中心（NLU事件分类）、赣政通（任务推送）、电子地图（人员位置）
- **核心价值：** 派单准确率从60%提升至95%，人工干预仅需确认
- **数据流向：** 调度指令→平台事件中心记录→被调度人员通过平台（赣政通/融合通讯）接收指令→处置结果回传平台。**平台是调度指令的唯一记录系统**
- **授权主体：** 政法委综治中心（日常调度规则），城管局和环保局（确认本部门调度规则）

**场景二：应急资源"一张图调度" —— "应急管理局授令，全资源一张图"**

> 山林火灾。应急管理局确认"进入应急模式"——调度权限升级，范围从网格员扩展至所有应急资源。GIS地图以火点为中心展示三个同心圆资源分布，系统推荐最优方案，应急管理局指挥员确认后一键下发——包括网格员（政法委管理）、护林员（林业局管理）、消防车（消防支队管理）、无人机。应急结束后退出应急模式，权限自动恢复。全程记录存档可审计。

- **协同模块：** 电子地图（同心圆资源展示）、无人机（自动起飞+火势分析）、融合通讯（一键调度）
- **核心价值：** 应急资源调度从"凭印象喊人"升级为"系统算出最优解+应急管理局授令"
- **数据流向：** 应急模式下所有调度指令和资源调配数据→平台事件中心留痕。应急结束后，平台自动生成应急响应报告。**平台是应急数据的唯一汇聚点**，应急管理局行使指挥权
- **授权主体：** 应急管理局（《突发事件应对法》法定应急指挥权）

**场景三：网格员"负荷均衡" —— "政法委综治中心规则下的自动优化"**

> 早上9:30，第2网格小张手头5个待办（饱和），相邻第3网格小李只有1个。第2网格新增"走访独居老人"任务。系统根据政法委综治中心预设的"同一街道内网格员可互助"规则，自动将任务改派给小李——备注"综治中心负荷均衡规则自动调整"。全程在政法委授权规则下运行，不跨部门。

- **协同模块：** 事件中心（工单管理）、电子地图（人员位置+负荷看板）、赣政通（任务推送）
- **核心价值：** 负荷均衡化，任务完成时效提升25%
- **数据流向：** 负荷均衡调整记录→平台事件中心留存→作为网格员工作量统计依据。**平台是工作负荷数据的唯一来源**
- **授权主体：** 政法委综治中心（网格员互助规则）

### 4.4 基层治理效能评估系统（新增模块）

#### 功能描述

基层治理效能评估系统是**基层工作的"数字体检师"**，帮助网格员发现自己工作的短板、帮助街道找到管理中的薄弱环节、帮助部门看清改进的方向。它从数据中台自动采集真实运行数据，不做排名惩罚，只做问题诊断和改进建议。核心定位：**"帮助基层发现问题、改进工作"，而非"给基层增加一个考核工具"。**

**政策依据：** 中央办公厅《关于解决形式主义突出问题为基层减负的通知》明确要求"改进督查检查考核办法"；中央政法委市域社会治理现代化试点要求"用数据说话、用数据管理、用数据决策"。

**与传统考核的本质区别：**

| 维度 | 传统考核 | 本系统 |
|------|---------|--------|
| 数据来源 | 基层填表上报，可能"美化" | 系统直接从平台取数，不可篡改 |
| 核心目的 | 排名→奖惩 | 诊断→改进 |
| 更新频率 | 每月/每季度一次，滞后 | 实时计算，随时可查 |
| 个人排名 | 公开个人排名（可能引发队伍不稳定） | 仅个人纵向对比（"和自己比进步"），不公开个人排名 |
| 组织排名 | 凭印象 | 红黑榜仅限街道/部门级（政府部门排名是常规管理手段） |
| 结果呈现 | 一个分数 | 五维雷达图+趋势曲线+AI改进建议 |

**核心设计原则（关键）：**

1. **不公开个人排名** —— 网格员只能看到自己的效能数据和历史对比，看不到其他人的排名。避免因排名引发队伍不稳定
2. **部门排名仅限内部** —— 红黑榜在部门负责人范围内可见，不对外公开
3. **核心指标是"减负成效"** —— 最重要的指标是"本系统帮你省了多少填表时间"，而不是"你完成了多少任务"
4. **每条建议都可操作** —— AI改进建议必须附具体的操作入口（点击跳转），而不是笼统的"请加强管理"

**核心能力清单：**

| 能力 | 说明 |
|------|------|
| 五维效能画像 | 任务完成率、巡查覆盖率、事件处置时效、群众满意度、数据鲜活度（权重由政法委统一配置） |
| 个人纵向对比 | 网格员只能看自己的——本月 vs 上月、本季度 vs 上季度，和自己比进步，不公开他人数据 |
| 街道/部门排名 | 街道级和部门级排名，红黑榜在部门负责人范围内可见，作为管理参考而非奖惩依据 |
| 基层减负成效可视化 | 统计"本系统帮你省了多少填表时间""自动生成了多少份报告"——让基层看到减负的实际好处 |
| 智能诊断报告 | AI根据薄弱指标自动生成改进建议，每条建议附具体操作入口（"点击这里开始使用AI眼镜AR引导检查清单"） |
| 数据下钻溯源 | 每个分数可下钻至原始工单——从"处置时效偏低"下钻至"3个超时工单详情+原因分析" |

#### 三个最常用场景（结合基层平台）

**场景一：网格员"自我体检" —— "不用等领导说，系统告诉我哪里还可以更好"**

> 网格员小李打开平台个人中心的"效能体检"页面，看到自己的五维雷达图——任务完成率98%（优秀）、巡查覆盖率95%（优秀）、事件处置时效88分（良好）、群众满意度96%（优秀）、数据鲜活度94%（优秀）。系统提示："相比上月，您的处置时效提升了6分，进步明显！"同时给出3条改进建议：①"您的3个超时工单均为'邻里纠纷调解'类，建议使用AI案例推荐辅助调解方案 → [点击试用]"；②"您负责的第3网格XX小区人口更新率偏低，建议本周安排一次入户走访 → [创建走访任务]"；③"您本月的AI眼镜使用率仅35%，低于街道平均的62%，建议在安全巡检场景启用 → [查看AI眼镜使用指南]"。小李点击第一条建议，直接跳转到AI案例推荐功能。他看不到其他网格员的分数——只有自己在和自己比。

- **协同模块：** 数据中台（全量运行数据）、AI赋能中心（智能建议生成）、大数据分析（趋势分析）
- **核心价值：** 从"被考核的压力"变为"被帮助的感觉"，改进建议可一键执行

**场景二：街道"红黑榜"（内部管理参考） —— "排名靠后不可怕，不知道为什么才可怕"**

> 每月5日，区综治中心的数据大屏自动更新各街道的治理效能综合得分。前三名进入红榜，末三位进入黑榜（仅街道办主任和区领导可见，不对下公开）。点击排名末位的"YY街道"——系统自动标注拖分项：事件处置时效偏低（65分，全区平均82分）、消防检查漏项率偏高（18%，全区平均6%）。下钻到原始数据：60%的超时事件集中在3个网格，均为消防安全检查类。系统进一步分析根因——这3个网格旧城改造密集、人均管理商户数是其他街道的1.5倍。自动建议：①为该街道增配2名网格员；②优先在这3个网格部署智能烟感（点击查看IoT传感器部署方案）；③消防检查清单从38项精简至22项。区领导不是只看"谁倒数"，而是看到了"为什么倒数、怎么帮他们改进"。

- **协同模块：** 大数据分析（数据下钻+根因分析）、AI赋能中心（改进建议生成）、电子地图（空间分布分析）、智慧调度（人员负荷计算）
- **核心价值：** 排名数据服务于"精准帮扶"，而非"简单奖惩"

**场景三：基层减负成效"看得见" —— "这个系统到底帮我省了多少事？"**

> 网格员每月5日收到一条赣政通消息："【您的月度减负报告】本月，基层智治平台为您：自动生成巡查日志38份（节省约6小时）、自动生成走访记录15份（节省约3小时）、AI帮您填写事件简报22份（节省约4小时）。3项合计节省约13小时的文书工作时间。您本月的个人效能画像已更新，点击查看 →" 这个功能让网格员直观感受到——平台不是来考核你的，是来帮你的。区领导也可以通过大数据平台看到全区的"减负成效总览"——本月全区基层人员累计节省文书工作时间约8,200小时，相当于释放了4名全职人员的产能。

- **协同模块：** 数据中台（运行数据统计）、AI赋能中心（自动生成报告数量统计）、赣政通（推送月度减负报告）
- **核心价值：** "减负成效可视化"是评估系统最重要的指标，让基层从"被考核"转向"被帮助"

## 五、第二个"1"：AI赋能中心（AI）—— 基层治理AI工作台

### 5.1 核心理念

> **不是给基层"AI功能"，而是给每个基层人员配一个"AI数字同事"。**
>
> 这个数字同事会听、会看、会写、会查、会提醒——它嵌入基层工作的每个环节，越用越聪明。基层不需要理解AI，只需要感受到"有人帮我干活了"。

### 5.2 架构：三层AI体系

```
+--------------------------------------------------+
|              第三层：AI运营中心                    |
|    (持续学习：人工修正→自动标注→模型迭代)           |
|    (质量看板：每个Agent的准确率、减负时长实时可见)    |
+--------------------------------------------------+
|              第二层：专题Agent工厂                  |
|   +--------+ +--------+ +--------+ +--------+ +--------+
|   | 采集   | | 巡查   | | 调解   | | 分拨   | | 预判   |
|   | Agent  | | Agent  | | Agent  | | Agent  | | Agent  |
|   +--------+ +--------+ +--------+ +--------+ +--------+
|   每个Agent = 专题知识库 +专业skill+ 场景工作流 + 决策模型     |
+--------------------------------------------------+
|              第一层：AI能力底座                    |
|   大模型(LLM) | 语音(ASR) | OCR | 视觉(CV)        |
|   (政务云本地化部署，数据不出域)                     |
+--------------------------------------------------+
```

**关键设计决策：**

| 决策 | 说明 |
|------|------|
| **"1个场景1个Agent"** | 不追求全能AI，追求场景专用AI。每个Agent只做一件事，做到极致 |
| **"先做减法再做大"** | 第一阶段只上线3个Agent（采集、巡查、预判），跑通闭环后再扩展 |
| **"运营重于建设"** | AI上线后的持续优化投入不低于建设投入的50%。AI是"活"的系统，不是一次性工程 |
| **"人在回路、AI兜底"** | AI建议始终需人工确认，但当AI发现紧急风险（火灾预警、独居老人异常）时，自动触发工单不等人 |

### 5.3 五个专题Agent

#### Agent 1：采集Agent —— "口述即录入"

| 维度 | 说明 |
|------|------|
| **核心能力** | **双向对话式采集**：AI主动追问缺失字段——"身份证号是多少？""该户是否有出租？"，像老网格员带新人一样引导式采集 |
| **杀手功能** | **"静默采集"模式**：AI眼镜全程录音，AI后台自动提取关键信息填入表单，网格员无需任何手动操作，做到真正的"零录入" |
| **解决痛点** | 基层录入慢、易出错、老人不会用APP。从"填表"变"聊天"，采集门槛降为零 |
| **减负效果** | 单次采集从15分钟降至3分钟（效率提升80%），新人第一天就能独立采集 |
| **技术支撑** | ASR语音转写 + NLU语义理解 + 对话状态管理 + LLM信息提取 |
| **落地阶段** | **第一阶段**（技术成熟度最高，ASR+NLU已是成熟技术，3个月内可上线） |
| **赋能N层模块** | AI眼镜（硬件终端）、二维码门牌（地址锚定）、数据中台（数据写入） |

#### Agent 2：巡查Agent —— "AR眼睛+AI大脑"

| 维度 | 说明 |
|------|------|
| **核心能力** | **"会学习的检查员"**：AI实时分析眼镜画面——看到灭火器自动识别有效期、看到电线自动判断是否私拉、看到通道自动计算宽度是否达标 |
| **杀手功能** | **"隐患知识图谱"**：将消防法规、建筑规范、历史隐患案例构建为知识图谱，AI理解"什么是隐患"而不仅仅是"看到了什么"。新发现的隐患类型自动入库，越查越准 |
| **解决痛点** | "新员工不会查、老员工漏项查"，隐患识别标准不统一 |
| **减负效果** | 单次检查从40分钟降至15分钟，漏项率从15%降至2%以下，隐患发现率从60%升至85% |
| **技术支撑** | CV视觉识别 + 知识图谱推理 + AR增强现实 + 端侧AI推理（离线可用） |
| **落地阶段** | **第一阶段**（与采集Agent同步上线，复用CV+知识图谱基础设施） |
| **赋能N层模块** | AI眼镜（画面采集）、无人机（航拍图像AI识别）、IoT传感器（告警联动校验） |

#### Agent 3：调解Agent —— "随身法律顾问"

| 维度 | 说明 |
|------|------|
| **核心能力** | 现场调解时，AI实时推送"三件套"——①相似案例及处置结果 ②适用法规条文（精确到第几条） ③推荐调解策略（"建议先安抚情绪，再谈赔偿"） |
| **杀手功能** | **"调解模拟器"**：新人培训时，AI扮演"上访群众"，网格员练习调解技巧，AI评判表现并给出改进建议。这个功能领导汇报时极其震撼——"我们用AI训练网格员如何化解矛盾" |
| **解决痛点** | 基层人员法律知识薄弱、调解靠经验、新人不敢独立调解 |
| **减负效果** | 调解成功率提升30%，新人独立调解培训周期从3个月缩短至2周 |
| **技术支撑** | RAG检索增强生成 + 语义向量索引 + 案例推理(CBR) + 对话系统 |
| **落地阶段** | **第二阶段**（需要积累3个月以上的案例库和政策库数据，建议采集Agent跑起来后再上线） |
| **赋能N层模块** | 事件闭环处置、赣政通（案例推送）、效能评估（调解成功率统计） |

#### Agent 4：分拨Agent —— "秒级精准派单"

| 维度 | 说明 |
|------|------|
| **核心能力** | **"预判式分拨"**：不仅分拨当前事件，还自动关联历史同类事件——"该地址过去3个月有4次同类投诉，建议升级处置""该类型事件本周已是第8起，可能存在系统性原因" |
| **杀手功能** | **"协同建议"**：AI不是简单派给一个人，而是推荐最佳协同组合——"建议城管主办+环保协办，参考XX街道类似事件的成功处置方案（处置时效18小时，满意度95%）" |
| **与智慧调度的分工** | 分拨Agent是"AI大脑"做决策建议（该派给谁），智慧调度系统是"执行系统"做派单和跟踪（派出去、跟回来）。两者各司其职 |
| **减负效果** | 派单准确率95%以上，人工干预减少80%，决策时间缩短60% |
| **技术支撑** | NLU意图识别 + 实体提取 + 地址匹配（一标三实） + 规则引擎 + 协同过滤推荐 |
| **落地阶段** | **第二阶段**（与调解Agent同步，需要历史工单数据训练分拨模型） |
| **赋能N层模块** | 智慧调度（执行派单）、赣政通（任务推送）、E呼即办（工单同步） |

#### Agent 5：预判Agent —— "治理天气预报"

| 维度 | 说明 |
|------|------|
| **核心能力** | **"治理天气预报"**：像天气预报一样预测治理风险——"下周XX街道空调噪音投诉预计上升30%，建议提前介入""下月XX区域进入汛期，历史数据显示该区域积水工单会增加50%" |
| **杀手功能** | **"风险热力预报"**：在GIS地图上展示未来7天的风险预测热力图（橙色=风险上升、红色=高风险），领导看大屏时像看天气预报一样看治理风险。**这个可视化效果在领导汇报时极其震撼** |
| **解决痛点** | "问题出了才知道""治理靠经验而非数据""预防性治理手段缺失" |
| **远期价值** | 从"接诉即办"升级为"未呼先办"，最终形成可对外销售的"城市治理风险预测"数据产品 |
| **技术支撑** | 时序分析 + 空间聚类 + 因果推断 + LLM辅助解读 + GIS可视化 |
| **落地阶段** | **框架先行**——历史数据积累3个月后模型即可上线。第三阶段正式发布预测功能 |
| **赋能N层模块** | IoT传感器（监测数据输入）、无人机（预防性巡飞触发）、智慧调度（预防性任务生成）、效能评估（预判准确率统计） |

### 5.4 AI运营中心（核心差异化能力）

> 这是当前方案完全缺失的一层。AI不是部署完就完事了——**运营重于建设。**

| 能力 | 说明 |
|------|------|
| **人工修正→自动学习** | 每次网格员修改AI建议（如修改了AI推荐的派单对象、修改了AI生成的简报内容），自动记录为训练样本。同一场景修改超过3次自动标记为"需优化"，触发模型迭代 |
| **Agent质量看板** | 每个Agent实时展示三项核心指标：①准确率（AI建议被采纳的比例） ②采纳率（AI建议不经修改直接使用的比例） ③减负时长（累计节省多少人力时间）。看板在领导大屏上实时可见 |
| **知识库自动更新** | 新政策法规发布→AI自动解析结构化→纳入知识库。历史案例结案→自动归档为训练样本。知识库永远保持"最新版本"，无需人工维护 |
| **AI"闭眼"机制** | Agent准确率低于85%时，自动降级为"仅建议、不自动执行"模式；低于70%时自动暂停该Agent，人工介入优化后重新上线。确保AI"不行的时候不添乱" |
| **减负成效可视化** | 全区驾驶舱实时展示："本月AI为全区基层人员节省X小时"，让AI价值量化、可见、可考核 |

### 5.5 实施路径

| 阶段 | 时间 | 上线Agent | 关键投入 | 验收标准 |

| **第一阶段** | 0-3个月 | 采集Agent + 巡查Agent + AI运营中心基础框架 | 1-2名AI工程师 + 政务云GPU服务器（2台） | 采集效率提升80%，巡查漏项率<5%，AI采纳率>85% |

### 5.6 人机协同原则

- **业务决策权永远在一线人员手中** —— AI提供建议，人做决定
- **AI建议必须"可解释"** —— 每条建议附带依据（相似案例编号、法规条文出处、数据来源），基层人员知道"AI为什么这么建议"
- **紧急风险AI可自动触发** —— 火灾预警、独居老人异常等时效性场景，AI不经人工确认自动生成工单，事后审计
- **每次人工修正都是AI的学习机会** —— "越用越聪明"不是口号，是系统设计

#### 6.3.2 基层信息采集模块

**模块定义：** 以标准地址为锚点，对"人、房、企、物"等治理要素进行新增、修改、删除、查询的全生命周期管理。

**核心动作：** 增、删、改、查（CRUD）

**子功能范围：** 人口采集、房屋采集、通用场所采集、地图标点/图斑采集、从业人员登记、重点人员标注、关注群体标记、出租房管理、标签管理、业户商户上报、自主上报、走访记录、楼栋管理

**匹配条件：**

- **用户要素：** 网格员（可采集辖区内所有类型）、民警/派出所人员（人口/房屋/重点人员）、综治工作人员（人口/场所）、物业人员（仅本小区房屋和人口）
- **地址要素：** 必须绑定有效标准地址。扫码获取为最高优先级，其次为任务单带入、手动选择地址树、历史记录回填。无标准地址不能进入采集表单。
- **业务意图：** 新增（需查重校验）、修改（仅限本辖区数据）、删除/注销（未被其他业务引用方可删除）、查询（按条件检索已采集数据）
- **前置状态：** 新增时目标对象在当前地址下不存在；修改时目标对象已存在且用户有修改权限；删除时目标对象未被其他业务引用；批量操作时逐条校验权限
- **设备要素：** 智能手机（必需），AI眼镜（可选增强）

**AI眼镜协同分析 —— ✅ ⭐⭐⭐ 推荐**

| 维度 | 评估 |
|------|------|
| 核心增益 | ①身份证/营业执照OCR→免手动录入 ②语音录入替代手打 ③拍照免掏手机 ④扫码门牌快2-3秒 |

| 不适合的情况 | ①室外嘈杂环境(>75dB) ②地图标点/图斑采集（需大屏操作） ③群众自主上报（群众无眼镜设备） |

#### 6.3.3 巡查轨迹管理模块

**模块定义：** 记录基层工作人员在物理空间中移动巡查的轨迹、时间、打卡动作及途中发现的问题，形成可追溯的巡查记录。

**核心动作：** 开始巡查 → 移动 → 打卡 → 发现 → 上报 → 结束巡查

**子功能范围：** 巡查路线、巡更打卡、护学打卡、任务巡查、走访日志

**匹配条件：**

- **用户要素：** 网格员（日常巡查）、民警/辅警（治安/夜间巡逻）、综治巡逻员（街面秩序）、护学岗人员（校门口巡查）、河长/林长（河道/山林巡查）
- **地址要素：** 必须现场扫码获取或地理围栏自动触发，确保"人到现场"，防止虚假打卡。GPS定位进入预设范围时自动确认。
- **业务意图：** 开始巡查（启动轨迹记录，关联巡查计划）、到达打卡（扫码/地理围栏确认到岗）、发现问题（快速拍照/语音简易上报）、结束巡查（停止轨迹，生成巡查日志）
- **前置状态：** 当前时间在巡查排班计划的时间范围内；打卡地址在巡查人员的管辖区内（允许非管辖区打卡，系统标记"跨区"并通知辖区负责人）；同一打卡点5分钟内不可重复打卡
- **设备要素：** 智能手机（GPS必须开启），AI眼镜+蓝牙指环（可选增强）

**AI眼镜协同分析 —— ✅✅ ⭐⭐⭐⭐ 强烈推荐**

| 维度 | 评估 |
|------|------|
| 核心增益 | ①免掏手机自动打卡 ②GPS轨迹自动记录 ③发现问题上报考勤"掏手机5步"降为"语音1步" ④巡查日志AI自动生成 |
| 最佳使用环境 | 户外徒步巡查，GPS信号良好，噪音<70dB的次干道和小区 |
| 不适合的情况 | ①夜市/主干道嘈杂(>80dB)→语音识别失效 ②中大雨天气 ③夜间光线不足拍照质量差 |
| 关键约束 | ①纯轨迹记录续航4-5h，间歇拍照3-4h，需外挂电池 ②GPS依赖手机定位 ③高温(>35°C)佩戴舒适度下降 |
| 失效降级 | 蓝牙断连→眼镜本地缓存，重连后同步；GPS信号弱→提示扫码手动打卡 |

#### 6.3.4 事件闭环处置模块

**模块定义：** 对矛盾纠纷、投诉举报、安全隐患等各类社会治理事件进行"发现→登记→分拨→处置→结案"的全流程闭环管理。

**核心动作：** 登记 → 分拨 → 处置 → 结案（全生命周期）

**子功能范围：** 矛盾纠纷、投诉举报、事件处置、安全隐患处置、简易矛盾处置、E调解、事件核实、人员核实、待办核实、个人上报、统战上报

**匹配条件：**

- **用户要素：** 网格员（登记、处置简易事件）、民警（登记、处置治安类事件）、社工（登记、参与调解）、物业人员（登记物业相关事件）、城管（登记、处置街面秩序类事件）、群众（仅可上报，不可处置）
- **地址要素：** 必须绑定标准地址（事发地址）。扫码获取或手动从地址树选择。事件的地址是"事发地"，不一定是"数据归属地"。
- **业务意图：** 新建事件（登记→待分拨）、分拨（指派→待处置）、处置（到场记录→待审核）、结案/驳回（确认处置结果/退回重新处置）、核实（对上报内容真伪核实）
- **前置状态：** 新建时该地址无同类型+同内容的未结案事件；处置时状态必须为"待处置"且用户为被指派人或同部门人员；结案时处置记录完整（含过程+照片+结果）
- **设备要素：** 智能手机（必需），AI眼镜（可选，按子场景区分）

**AI眼镜协同分析 —— ⚠️ ⭐⭐ 选择性推荐（按子场景区分）**

事件处置的子场景差异极大，必须区分对待：

| 场景类型 | 推荐程度 | 理由 |
|---------|---------|------|
| 事故现场勘查（交通/安全生产） | ✅ 推荐 | 第一视角录像固定证据链，解放双手边测量边记录 |
| 安全隐患实地勘查 | ✅ 推荐 | AR标注隐患位置，远程专家实时指导 |
| 静默调查取证（暗访/蹲点） | ✅ 推荐 | 隐蔽拍照不惊动对象 |
| 邻里纠纷现场调解 | ❌ 不推荐 | 争吵噪音75-90dB→ASR失效；录像眼镜破坏信任氛围 |
| 家庭矛盾调解 | ❌ 不推荐 | 佩戴录像设备破坏信任，HUD弹出信息分散注意力 |
| 群体性事件处置 | ❌ 不推荐 | 设备损坏风险+应专注态势感知 |
| 领导接访 | ❌ 不推荐 | 面对面沟通不宜戴录像设备 |

#### 6.3.5 业务审核模块

**模块定义：** 对用户提交的各类申请（房屋绑定、居住证、低保、补贴等）进行"审核→通过/驳回"的审批操作，构成业务办理的控制节点。

**核心动作：** 查看申请 → 核实 → 通过/驳回（写审核意见）

**子功能范围：** 房屋绑定审核、人口审核、居住证申请审核、场所审核、消防自查审核、反诈上报审核、校园审核等12项

**AI眼镜协同分析 —— ❌ ⭐ 不推荐**

业务审核是桌面审批工作流，核心操作是阅读材料+点击按钮，手机/PC端体验远优于眼镜小屏。唯一例外是需"现场核实"的审核（如适老化改造对象认定、危旧房现场评估），但现场核实环节本身归属信息采集(6.3.2)或安全检查(6.3.6)模块。**结论：不在审核环节配AI眼镜。**

---

#### 6.3.6 安全巡检与宣防模块

**模块定义：** 按预设检查清单（SOP），对场所、设施、人员进行逐项安全检查，记录检查结果（合格/不合格/不适用），拍照留证，生成结构化检查报告。

**核心动作：** 选择检查清单 → 逐项检查 → 拍照 → 判定（合格/不合格/不适用） → 生成报告

**子功能范围：** 场所消防检查、消防自查、场所检查、反诈宣传/预警/劝阻、商户检查、重点安保单位检查、工矿企业检查、校园安全检查等11项

**AI眼镜协同分析 —— ✅✅✅ ⭐⭐⭐⭐⭐ 最强烈推荐（100%适配，17/17项支持）**

**这是AI眼镜的杀手级场景。** 安全检查的三大痛点与眼镜三大能力完美对应：

| 痛点 | 眼镜方案 |
|------|---------|
| "新员工不会查、老员工漏项查" | AR逐项引导检查清单，不跳过任何一项 |
| 人工肉眼判断隐患，标准不统一 | AI识别隐患（电线私拉、通道堵塞、灭火器失效），统一判定标准 |
| 检查完回办公室整理照片+写报告耗时15分钟 | 检查结束即生成结构化报告，耗时0分钟 |

| 子维度 | 详细分析 |
|--------|---------|
| 效率增益 | 单次检查从40分钟降至15分钟（减少62.5%），报告生成从15分钟→0分钟 |
| 质量标准 | 漏项率从15%降至<2%（减少87%），隐患发现率从60%升至85% |
| 最佳环境 | 商铺/餐馆/企业/学校等固定场所，室内或半室内，噪音<75dB |
| 不适用环境 | 化工厂防爆区（非防爆认证）、高粉尘环境（面粉厂/木材加工）、冷库<-18°C |
| 电池策略 | AR引导+间歇拍照模式续航3-4h，配外挂胶囊电池延长至7-9h |
| 离线能力 | 地下室/偏远工厂无网络→端侧AI离线检查→回网后自动同步 |

**与巡查轨迹模块(6.3.3)的关键区别：**

| 维度 | 巡查轨迹(6.3.3) | 安全检查(6.3.6) |
|------|-----------------|-----------------|
| 核心行为 | 移动中的轨迹记录+路过式观察 | 到达后的定点逐项检查 |
| 结构化程度 | 低（发现问题随口上报） | 高（按预设清单逐项判定） |
| 产出物 | 轨迹地图+巡查日志 | 结构化检查报告+整改通知书 |
| 时间特征 | 边走边记，持续1-3小时 | 到达后停留15-40分钟 |
| AI眼镜价值 | 免掏手机、自动打卡 | AR引导+AI识别+自动报告（杀手场景） |

#### 6.3.7 便民申请模块

**模块定义：** 面向群众（或网格员代为操作），提供物业缴费、居住证申请、维修报修、事件上报、问卷调查等民生服务的一站式入口。

**核心动作：** 选择服务 → 填写信息 → 提交申请 → 查看进度

**子功能范围：** 物业缴费、房屋维修、家具维修、便民热线、居住证申请、积分交易、事件上报、问卷调查、投票选举、闲置物品发布（H5页面跳转）

**AI眼镜协同分析 —— ❌ ⭐ 不推荐**

以窗口服务、表单填写、线上支付为主，核心操作均在手机屏幕完成。眼镜HUD无法展示复杂表单，也无法完成支付等交互。全部18个便民服务功能点均不推荐AI眼镜。

#### 6.3.8 任务执行反馈模块

**模块定义：** 接收系统下发的具体工作任务（走访、核查、检查），按任务要求到达指定地址、完成指定动作、记录执行结果并提交。

**核心动作：** 接收任务 → 查看要求 → 前往地址 → 签到 → 执行 → 提交结果

**子功能范围：** 数据任务、走访任务、场所巡查、业户走访、走访记录、走访评估、走访统计

**AI眼镜协同分析 —— ✅ ⭐⭐⭐ 推荐（10/13功能点推荐，77%）**

| 维度 | 评估 |
|------|------|
| 核心增益 | ①任务导航减少找地址时间 ②到场自动扫码确认（零操作） ③语音记录替代手写走访记录 ④多任务自动切换+路线优化 |
| 不适用情况 | ①纯政策宣传类任务（不需要复杂记录） ②涉高度敏感人员（戒毒/精神障碍）的走访 |
| 关键约束 | ①任务推送用"静默通知"模式（HUD角标+轻微振动），避免频繁弹窗干扰 ②导航依赖手机GPS，城中村等密集区域精度可能不足 |
| 失效降级 | GPS信号弱→提示"接近目标50米，请寻找门牌扫码确认" |

**与信息采集(6.3.2)的关键区别：**

| 维度 | 信息采集(6.3.2) | 任务执行(6.3.8) |
|------|-----------------|-----------------|
| 触发方式 | 自主发起 | 任务驱动 |
| 地址获取 | 可扫码/手动选/任务带入 | 必须扫码且地址完全一致 |
| 灵活性 | 高（自由选择采集对象） | 低（按任务要求执行） |
| 考核关联 | 无直接考核 | 有完成率、时效等考核 |

#### 6.3.9 数据展示模块

**模块定义：** 以标准地址为锚点，以可视化形式（驾驶舱、报表、图表、分色图）综合展示该地址相关的多维度数据，为决策提供数据支撑。

**核心动作：** 选择地址 → 加载数据 → 查看图表/报表（只读操作）

**子功能范围：** 数据驾驶舱、楼栋信息展示、房源信息展示、分色图、通知公告

**AI眼镜协同分析 —— ❌ ⭐ 不推荐**

数据展示需要较大屏幕展示图表、地图、表格，眼镜HUD显示区域有限（等效3米外40寸屏），不适合复杂数据浏览。唯一弱关联：到达楼栋时HUD浮现关键摘要（如"本楼3户重点关注"）——属路过式信息提示，可随巡查/采集场景附带实现。

#### 6.3.10 用户账户与绑定管理模块

**模块定义：** 管理用户个人账户信息，包括登录、资料编辑、角色切换、房屋绑定、家庭成员关联等。

**子功能范围：** 前台登录、编辑资料、我的家人、我的房屋、维修申请单、我的小区、我的车辆、角色切换、绑定房屋、修改密码、退出登录、陌生人员信息

**AI眼镜协同分析 —— ❌ ⭐ 不推荐**

纯后台配置和个人设置操作，无现场作业属性，与AI眼镜使用场景无关。

#### 6.3.11 通用辅助操作模块

**模块定义：** 提供跨模块通用的工具性功能，如全局搜索、二维码扫描、区域切换、主题切换等。

**子功能范围：** 切换区域、切换列表/数据/主题色、无障碍功能、扫一扫、搜一搜

**AI眼镜协同分析 —— ⚠️ ⭐⭐ 选择性推荐**

仅两个功能推荐眼镜：二维码扫描（比掏手机快2-3秒）和语音搜索（"搜XX路12号"直接出结果）。切换区域/主题色/无障碍等功能与眼镜无关，不推荐。
