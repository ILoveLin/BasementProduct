# "1+N+1+N" (PFAS) 总体架构功能说明

> 架构定义：1个基础平台（Platform） + N个生态平台（Federation） + 1个AI赋能中心（AI） + N个治理场景（Scenarios）

---

## 架构总览

"1+N+1+N" 是一个分层解耦、能力复用的四级架构体系。底层提供共性能力，中间层引入外部生态和AI智慧，顶层面向业务场景灵活组合，实现"厚底座、强智能、活生态、多场景"的技术格局。

四层之间的关系：

- **基础平台**是"地基"——所有能力的统一承载层
- **生态平台**是"插件"——引入外部优秀产品与政务系统的联接层
- **AI赋能中心**是"大脑"——贯穿各层的智慧注入层
- **治理场景**是"屋顶"——面向一线用户的业务呈现层

```
N个治理场景  ← 面向角色、面向业务、灵活组合
    ↕
1个AI赋能中心 ← 贯穿上下、注入智能
    ↕
N个生态平台  ← 横向扩展、外部联接
    ↕
1个基础平台  ← 纵向支撑、共性沉淀
```

## 一、1个基础平台（Platform）—— 共性能力中台

### 1.1 定位

基础平台是整个体系的**统一底座**，由公司自主研发，承担"能力沉淀、数据汇聚、标准统一、服务开放"四项核心职责。它是平台"厚底座"战略的物理载体——所有上层能力必须通过底座才能访问数据、调用服务、触达用户。

### 1.2 六大功能模块

#### 1.2.1 统一门户与身份认证

| 能力 | 说明 |
|------|------|

| 统一身份认证 | 对接江西省统一身份认证平台，支持人脸识别、数字证书等多因素认证 |

| 权限精细化管控 | 基于"角色+辖区+数据分级"的权限模型，网格员只能看本网格数据 |
| 多端适配 | 支持PC端、移动端（APP/小程序）、AI眼镜端统一登录 |

## 二、N个生态平台（Federation）—— 能力与联接延伸

### 2.1 定位

生态平台层是本平台与外部世界联接的桥梁。它遵循"开放对接、为我所用、不重新发明"的原则，既引入市场上优秀智能硬件的能力，又深度对接本地政务系统，让本平台不成为**新的孤岛**，而是现有政务生态的"**赋能枢纽**"和"**黏合剂**"。

**核心理念：** 每个模块既可以作为独立产品对外交付，也可以融入基层智治综合协同平台（第一个"1"），与其他模块组合形成完整的治理解决方案。所有模块采集的数据统一汇聚至基层平台数据中台，各部门按需同步。

### 2.2 数据主权与流向原则

```
采集层（AI眼镜/无人机/二维码门牌/IoT传感器/摄像头/北斗终端）
    ↓ 全部数据统一汇入
存储层（基层平台数据中台——"一标三实"唯一数据源）
    ↓ 按需同步/推送至各部门系统
使用层（公安人口库 / 消防监管系统 / 城管执法系统 / 民政养老系统 / ...）
```

| 原则 | 说明 |
|------|------|
| **平台是唯一数据源** | 所有采集数据统一存储在基层平台数据中台，不做分散存储 |
| **部门按需同步** | 部门从平台同步所需数据，或者按需平台推送原始数据 |
| **基层只录一次** | 网格员/民警/检查员在平台录入一次，多部门按权限复用 |
| **同步方向为"平台→部门"** | 数据流向始终是"基层平台→部门系统"，部门系统不直接向平台写入数据 |

## 三、1个AI赋能中心（AI）—— 平台级智慧大脑

### 3.1 定位

AI赋能中心不是孤立的AI功能模块，而是**贯穿各层的平台级智慧引擎**。它将大模型、OCR、语音识别等AI能力注入基础平台、生态平台和治理场景的每个环节，让知识"自动工作"，是近期基层减负最直接的抓手。

### 3.2 四大核心功能

#### 3.2.1 智能案例编写与推荐

| 维度 | 说明 |
|------|------|
| 能力描述 | 网格员语音简述事件经过 → AI自动匹配历史相似案例 → 生成简报初稿 → 人工确认发布 |
| 解决痛点 | 告别手写报告，改变"不会写、不愿写、写得慢"的现状 |
| 减负效果 | 每人每天节省约40分钟报告撰写时间 |
| 技术支撑 | 语音转写（ASR）+ 语义理解（NLU）+ 案例相似度匹配（向量检索）+ 文本生成（LLM） |

#### 3.2.2 知识库智能问答

| 维度 | 说明 |
|------|------|
| 能力描述 | 工作人员语音提问 → AI从政策法规、办事流程、历史案例知识库中检索答案 → 秒级返回 |
| 解决痛点 | 基层人员"不知道政策依据""不知道该找哪个部门""新人上手慢" |
| 减负效果 | 新人培训周期从1个月缩短至1周；现场政策查询正确率从60%提升至90% |
| 技术支撑 | RAG（检索增强生成）+ 语义向量索引 + 多轮对话 |

#### 3.2.3 制度库自动校验

| 维度 | 说明 |
|------|------|
| 能力描述 | 事件分拨、结案时AI自动比对制度库中的办理时限、材料格式、审批层级要求 → 不合规自动提醒 |
| 解决痛点 | 基层"超期不知情""材料不合规反复退件""审批越级或漏级" |
| 减负效果 | 同类事件处置一致性提升至85%；事件超时率大幅下降 |
| 技术支撑 | 规则引擎 + LLM语义理解 + 流程状态机自动化比对 |

#### 3.2.4 治理数据洞察与预判（远期）

| 维度 | 说明 |
|------|------|
| 能力描述 | 深度挖掘治理运行数据 → 自动发现规律 → 生成风险预测 → 提出干预建议 |

## 四、N个治理场景（Scenarios）—— 靶向解决一线痛点

### 4.1 定位

治理场景层是**面向角色、面向业务的复合应用层**。它不是独立的子系统，而是基于基础平台能力+AI赋能+生态平台能力，按"用户角色+核心业务"组合而成的综合解决方案。新增场景通过"五大要素判定法"快速匹配和开发。

### 4.2 四大核心场景

| 关键协同 | E呼即办（诉求联动）、AI赋能中心（月报生成）、数据中台（态势数据） |

### 4.3 场景扩展机制（五大要素判定法）

| 要素 | 判定内容 | 示例 |
|------|---------|------|
| 用户要素 | 谁在用？（角色+权限） | 河长 → 河道巡查场景 |
| 地址要素 | 在哪用？（标准地址来源） | 扫码河段二维码→获取河段编码 |
| 业务意图 | 要干什么？（操作类型） | 巡查→记录水位、拍照、上报污染 |
| 前置状态 | 能用吗？（数据/流程前提） | 当前时段在巡查排班计划内 |
| 设备要素 | 用什么设备？ | 手机必须（GPS）+ AI眼镜推荐 + 无人机备选 |

匹配到已有模块→直接复用；不匹配但有共性→抽象为新通用模块（N=5、N=6…）；无共性→独立开发。

## 五、四层之间的关系与数据流

### 5.1 纵向关系

```
治理场景（N）    ←  面向角色呈现，调用下层能力
    ↕ 调用/反馈
AI赋能中心（1）   ←  横贯各层：向上赋能场景、向下注入底座、左右联接生态
    ↕ 调用/反馈
生态平台（N）    ←  横向扩展：硬件能力接入 + 政务系统联接
    ↕ 调用/反馈
基础平台（1）    ←  纵向支撑：数据、GIS、通讯、流程、AI五个统一能力底座
```

### 5.2 关键数据流

| 流向 | 说明 | 示例 |
|------|------|------|
| 群众→平台 | 通过赣服通、二维码门牌、E呼即办三种入口提交诉求 | 扫码报修→生成工单→流入事件中心 |
| 平台→基层 | 赣政通任务推送、本平台APP消息通知 | 指挥中心派单→网格员收到"走访独居老人"任务 |
| 基层→平台 | 巡查采集数据、事件处置反馈 | AI眼镜拍照+语音上报→写入一标三实数据湖 |
| 平台←→生态 | 与外部平台双向数据同步 | E呼即办工单同步→本平台地址匹配→处置结果回传E呼即办 |
| AI贯穿 | 在各环节注入智能服务 | 语音→转写→语义分析→自动填单→案例推荐→月报生成 |

### 5.3 核心理念

> **"赋能而非替代"**——本平台不替代任何现有部门业务系统，只做"黏合剂"和"减压阀"。
> 让公安的系统继续管公安的数据，综治的流程继续走综治的审批。本平台做的是：用**标准地址**把各系统的数据串起来，用**AI**把重复劳动替掉，用**生态**把外部能力接进来。
