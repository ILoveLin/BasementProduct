# 《基层智治综合协同平台》产品方案书 V2.0 · 方案B（贯穿版）

> **版本说明：** 本版将**事件总线**和**安全与合规底座**都作为横向贯穿能力——事件总线是贯穿四层的"中枢神经"，安全底座是覆盖全平台的"免疫系统"。两者不隶属于任何一个单层，而是作为全平台的基础设施运行。1+N+1+N 四层结构完全保持不变。

---

## 核心架构决策：为什么选择"双贯穿"？

### 事件总线为什么不能只放在Platform？

| 维度 | 融入Platform（方案A） | 贯穿全平台（方案B） |
| --- | --- | --- |
| **理论一致性** | 事件总线被视为Platform的"输出能力" | 事件总线被视为"全平台运行范式"——不仅是Platform的能力，Ecology的感知、AI的分析、Scenarios的呈现都围绕事件流转展开 |
| **扩展性** | 新增N模块需通过Platform事件总线接入 | 新增N模块直接接入事件总线——事件总线是"公共管道"而非Platform的"私有管道" |
| **与AI层的关系** | AI通过Platform读取事件数据 | AI可以从事件总线直接订阅事件流，进行实时分析——预判Agent可以直接"监听"事件总线 |
| **汇报说服力** | "我们在Platform里加了事件引擎" | "整个平台是事件驱动的——每一层的运转都围绕事件流转展开"（对领导汇报更有冲击力） |

### 安全底座为什么也要贯穿？

| 维度 | 单独成层 | 贯穿全平台（方案B） |
| --- | --- | --- |
| **架构纯净性** | 增加一个"第零层"，破坏1+N+1+N结构 | 不增加层数，安全作为"横向属性"注入每一层 |
| **责任归属** | 安全是"安全团队的事" | 安全是"每层设计者的事"——Platform负责数据安全、AI负责Agent合规、Ecology负责设备合规、Scenarios负责场景合规 |
| **实际落地** | 安全要求写在一个独立章节 | 安全要求嵌入每个层级的描述中，评审时逐层check |

### 2.1 事件总线（横向贯穿——中枢神经）

> **定位：** 事件总线不隶属于Platform，而是作为贯穿全部四层的"公共基础设施"运行。它定义了整个平台的运转节拍——所有感知（Ecology）、分析（AI）、处置（Scenarios）都围绕事件的六步流转展开。

#### 事件总线与四层的关系

| 层级 | 与事件总线的关系 | 该层在事件流转中的角色 |
| --- | --- | --- |
| **Platform** | 提供事件总线运行所需的基础数据（地址、网格、部门映射） | **数据支撑者**——为分拨环节提供精准的地址匹配和网格定位 |
| **Ecology** | 所有N模块的感知数据统一发布到事件总线；所有N模块从事件总线接收工单 | **事件生产者+执行终端**——产生事件、接收工单、回传结果 |
| **AI** | AI Agent直接订阅事件总线数据流，进行实时分析和预测 | **智能分析者**——分拨Agent推荐分拨方案、预判Agent分析事件趋势 |
| **Scenarios** | 每个场景是事件总线的一个"流程模板"——预定义流转路径、参与部门、时效规则 | **流程定义者**——"消防检查一件事"就是在事件总线上运行的标准流程 |

#### 事件流转六步闭环

```
发现（Ecology感知层：AI眼镜/IoT/无人机/群众扫码）
  → 上报（事件总线：智能去重+结构化+地址关联）
  → 分拨（事件总线调用Platform规则引擎+AI分拨Agent辅助）
  → 处置（事件总线分发工单至Ecology执行端+AI调解Agent辅助）
  → 反馈（Ecology执行端回传结果至事件总线+群众回访）
  → 评价（事件总线汇总数据→效能评估系统分析）
```

### 2.2 安全与合规底座（横向贯穿——免疫系统）

> **定位：** 安全底座是覆盖全部四层的"免疫系统"——每一层都有自己要守护的安全边界，整体构成纵深防御体系。

| 层级 | 该层的安全职责 | 具体要求 |
| --- | --- | --- |
| **Platform** | **数据安全主责** | 政务云不出域；存储加密（SM4）+传输加密（TLS 1.3）；数据分级分类+访问权限最小化；操作日志全审计；数据脱敏 |
| **Ecology** | **设备/接入安全主责** | 终端设备认证（禁止未授权设备接入）；通信加密（IoT NB-IoT+应用层国密）；供应商安全评估（≥2家互备）；固件安全更新 |
| **AI** | **Agent合规主责** | Agent输出内容过滤；AI建议强制可溯源；闭眼兜底机制；关键决策强制人工确认；法规定期对齐 |
| **Scenarios** | **场景合规主责** | 个人信息保护（AI行为识别不采集个体身份）；特定场景额外合规（校园场景保护未成年人、养老场景保护特殊群体）；场景上线前合规评审 |

#### 全平台共性安全要求

| 维度 | 要求 |
| --- | --- |
| **信创适配** | 服务端支持鲲鹏/飞腾+麒麟/统信+达梦/人大金仓；客户端支持鸿蒙OS；终端优先国产化 |
| **等保密评** | 等保三级；密评（SM2/SM3/SM4应用）；24个月一次复评 |
| **安全运维** | 定期渗透测试+漏洞扫描；重大安全事件2小时内启动应急响应 |
| **法规遵从** | 《数据安全法》《个人信息保护法》《无人机飞行管理暂行条例》《突发事件应对法》持续合规跟踪 |

## 三、第一个"1"：基础平台（Platform）—— 共性能力中台（方案B版本）

> **相比方案A的重要区别：** Platform不承担"事件总线引擎"——它只负责提供事件总线运行所需的基础数据服务（地址匹配、网格定位、部门映射）。事件流转本身由横向贯穿的事件总线设施负责。

### 3.1 定位

基础平台是整个"1+N+1+N"体系的**统一数据底座**，承担"能力沉淀、数据汇聚、标准统一、服务开放"四项核心职责。它为事件总线提供**数据支撑**（地址解析、网格匹配、部门映射），但事件流转本身由横向贯穿的事件总线设施负责——**Platform是"数据提供者"而非"事件流转者"**。

### 3.2 四大共性能力

*（保留V1原有四大能力：数据中台与治理数据湖、时空地理信息GIS底座、周边、格友圈/协同圈——全部内容不变）*

### 3.3 Platform与事件总线的分工（新增）

| 能力 | 归属 | 说明 |
| --- | --- | --- |
| 地址解析与网格匹配 | **Platform** | 事件总线调用Platform的"一标三实"地址库进行地址→网格→责任部门的匹配 |
| 事件流转与状态管理 | **事件总线** | 事件从创建到评价的整个生命周期状态管理，独立于Platform运行 |
| 分拨规则存储 | **Platform** | 政法委综治中心预定义的分拨规则存储在Platform规则引擎中，事件总线调用 |
| 工单生成与分发 | **事件总线** | 事件总线根据分拨结果生成工单并分发至对应N模块（赣政通/融合通讯等） |

## 十、总结与展望（方案B版本）

"1+N+1+N"架构的核心逻辑是：

以**事件总线**为中枢神经（贯穿四层，驱动"发现→上报→分拨→处置→反馈→评价"的治理闭环），以**安全与合规底座**为免疫系统（贯穿四层，守护数据安全/Agent合规/信创/法规），以基层治理平台为数据底座（四大共性能力），以生态平台（分级分类：深度绑定/按需调用/预留扩展）为能力插件，以AI赋能中心（5个专题Agent，辅助决策、专题知识库+场景Skill）为智慧大脑，灵活支撑N个治理场景（含上饶专属场景：婺源文旅/红色乡村/信江生态）。

**一句话总结：** 事件驱动流转，安全护航全程，数据沉淀资产，场景构筑壁垒。
