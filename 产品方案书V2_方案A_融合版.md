## 〇、问题对标与预期目标（新增章节）

### 0.1 当前基层治理的核心问题对标

本平台精准对标本市基层治理的三组核心矛盾：

| # | 核心矛盾 | 具体表现 | 平台解决路径 | 可量化目标 |
| --- | --- | --- | --- | --- |
| **1** | **报表多头报送** | 同一数据（人口、房屋、隐患）需向公安、综治、住建、民政等4-6个部门分别填报，网格员每月填报时间占工作总量的35%以上 | "一标三实"统一数据底座→一次采集、多方复用；AI采集Agent口述即录入 | 网格员填报时间从35%降至10%以下，减少重复填报80% |
| **2** | **系统多但用不起来** | 各部门自建系统10+个，但数据不互通、流程不衔接、基层不愿用——"上面建了、下面晾着" | 不替代任何现有系统，做"赋能工具+协同枢纽"；二维码门牌统一入口、赣政通统一待办、AI眼镜降低使用门槛 | 基层日活使用率≥70%（当前三区县验证数据：62%） |
| **3** | **城乡二元结构诉求不同** | 城区核心诉求是"精细化管理"（安全隐患、矛盾纠纷、民生服务）；乡镇核心诉求是"底线民生"（独居老人关怀、防灾减灾、农房安全）——同一套系统不能"一刀切" | 城区场景：立体巡防、一码共治、消防安全；乡镇场景：独居老人关爱（北斗终端）、防溺水/防汛（无人机+IoT）、农房安全巡查；上饶专属场景深度适配 | 城区和乡镇各建成≥3个差异化标杆场景 |

## 二、总体架构图与功能说明（重要修订）

> "1+N+1+N"（PFAS）架构体系围绕基层治理"**数据怎么管、事件怎么转、能力怎么扩、智慧怎么用、问题怎么解**"五大核心问题。**新增：安全与合规底座（Security）横向贯穿全部四层，事件总线引擎（Event Bus）融入基础平台作为核心运转机制。**

### 2.0 安全与合规底座（新增——横向贯穿层）

> **定位：** 安全与合规底座不是1+N+1+N之外的"第五层"，而是四层架构的**横向贯穿基础**——每一层都建立在安全底座之上，每一层都受合规框架约束。

| 维度 | 覆盖范围 | 核心内容 |
| --- | --- | --- |
| **数据安全** | Platform层 | 政务云本地化部署不出域；数据分级分类（核心/重要/一般）；存储加密（SM4国密）+传输加密（TLS 1.3）；数据访问权限最小化；操作日志全审计；数据脱敏（敏感字段动态掩码） |
| **Agent生成内容合规** | AI层 | AI建议强制附带可溯源依据；关键决策节点人工确认（非自动执行）；Agent输出内容过滤（防范不当言论/错误引导）；"闭眼"机制——准确率<85%降级为仅建议、<70%自动暂停 |
| **信创适配** | Platform+Ecology层 | 服务端：支持鲲鹏/飞腾CPU + 麒麟/统信OS + 达梦/人大金仓数据库；客户端：支持鸿蒙OS（赣政通移动端）；终端设备：优先国产化方案（北斗终端已国产） |
| **等保密评** | 全层 | 等保三级（地市级政务平台最低要求）；密评（商用密码应用安全性评估）——SM2/SM3/SM4在数据传输和存储环节的应用 |
| **法规遵从** | 全层 | 《数据安全法》《个人信息保护法》（AI行为识别不采集个体身份）；《无人机飞行管理暂行条例》（UTMISS备案）；《突发事件应对法》（应急指挥授权）；江西省数字政府建设相关法规 |
| **安全运维** | 全层 | 定期渗透测试+漏洞扫描；24个月一次等保复评；安全事件应急响应SLA（重大事件2小时内启动） |

#### 3.2.5 事件总线引擎（新增——核心能力）

> **这是基层治理的本质运行逻辑。** 事件总线引擎定义了"一件事在平台上如何从头跑到尾"——它是连接数据底座（Platform）、感知终端（Ecology）、AI大脑（AI）和应用场景（Scenarios）的**中枢神经系统**。

##### 事件流转六步闭环

```
发现 → 上报 → 分拨 → 处置 → 反馈 → 评价
  │      │      │      │      │      │
  │      │      │      │      │      └── 群众/上级评价+效能自动评分
  │      │      │      │      └──────── 处置结果回传+群众回访
  │      │      │      └────────────── 多部门协同处置+全程留痕
  │      │      └──────────────────── 规则引擎+AI分拨Agent辅助
  │      └────────────────────────── 语音/扫码/自动传感多渠道
  └──────────────────────────────── AI眼镜/IoT/无人机/群众扫码
```

| 环节 | 核心能力 | 说明 |
| --- | --- | --- |
| **①发现** | 多源感知汇聚 | 人工发现（网格员巡查、群众扫码上报）+ 自动发现（IoT传感器告警、AI摄像头识别、无人机AI识别、北斗终端异常预警）→ 统一接入事件总线 |
| **②上报** | 智能去重+结构化 | AI自动去重（同一地址+同类事件不重复建单）；语音/图片/视频自动转写为结构化工单；自动关联标准地址和归口部门 |
| **③分拨** | 规则引擎+AI辅助 | 日常事件按政法委综治中心预设规则自动分拨；应急事件按应急管理局应急预案分拨；分拨Agent提供推荐建议，**人工确认后执行** |
| **④处置** | 协同处置+全程留痕 | 多部门在同一事件页面内协同；支持融合通讯在线会商；AI调解Agent实时推送法规和相似案例；处置过程时间轴记录、不可篡改 |
| **⑤反馈** | 处置结果回传+群众回访 | 处置结果含照片/视频证据回传至事件总线；群众满意度回访（自动短信/电话/扫码评价）；未达标自动退回重新处置 |
| **⑥评价** | 自动效能评分+趋势分析 | 基于处置时效、满意度、复发率等维度自动评分；数据接入效能评估系统；高频事件自动标记为"需制度性解决" |

##### 事件总线的三大设计原则

| 原则 | 说明 |
| --- | --- |
| **分权而非集权** | 不同场景的调度指令来自不同的法定授权主体（政法委综治中心→日常事件、应急管理局→应急事件、E呼即办→民生诉求），事件总线是"流转管道"而非"集权工具" |
| **全程可追溯** | 每个事件从发现到评价的完整时间轴、每个环节的操作人和操作内容、每次流转的耗时——全部记录，不可篡改 |
| **弹性扩容** | 日常流量和峰值流量（防汛/山火等应急期间的事件激增）——事件总线支持弹性扩展，峰值承载能力≥日常的5倍 |

## 五、第二个"1"：AI赋能中心（AI）—— 基层治理AI工作台（重要修订）

> **相比V1的核心变化：** ①强化"辅助决策"定位，明确"业务决策权在业务员侧"；②Agent具象化为"专题知识库+场景Skill"组合；③新增Agent合规控制。

### 5.1 核心理念（修订）

> **不是给基层"AI功能"，而是给每个基层人员配一个"AI数字同事"。**
>
> 这个数字同事**只做建议、不做决定**——它会听、会看、会写、会查、会提醒，但最终判断永远在人手里。
>
> **关键原则：AI做辅助决策，业务决策权永远在一线业务员侧。**

### 5.2 Agent的构成：专题知识库 + 场景Skill（新增）

> 每个Agent具象化为两个明确组件：
> - **专题知识库：** 该领域经过结构化处理的知识集合（法规、案例、操作手册、隐患图谱）
> - **场景Skill：** 该场景下的具体AI行为能力（语音转写、图像识别、案例匹配、趋势预测）

| Agent | 专题知识库 | 场景Skill | 辅助决策方式 |
| --- | --- | --- | --- |
| **采集Agent** | 地址标准库、人口/房屋/企业数据字典、字段校验规则库 | 语音→表单转写、OCR证件识别、对话式追问引导、静默采集 | AI填充表单→网格员确认→提交 |
| **巡查Agent** | 消防法规库、建筑规范库、历史隐患图谱、检查清单模板库 | 隐患视觉识别、AR检查项引导、自动生成检查报告 | AI标记隐患→检查员确认→生成报告 |
| **调解Agent** | 法律条文库（民法典/治安管理处罚法等）、相似案例库、调解策略模板库 | 相似案例检索、适用法规条文定位、调解策略建议、"调解模拟器"培训 | AI推荐法规+案例→调解员参考→自主判断 |
| **分拨Agent** | 部门职责矩阵、历史分拨记录、网格/人员负荷数据、应急分拨规则 | 事件类型自动识别、归口部门匹配、最佳处置人推荐、协同组合建议 | AI推荐分拨方案→综治中心确认→下达 |
| **预判Agent** | 历史事件时序库、空间聚类模型、季节性规律库、风险因子关联库 | 周期性预测、空间热力预测、高频问题根因分析、风险热力图生成 | AI预测风险→业务部门研判→制定预防措施 |

### 5.6 人机协同原则（修订）

### 6.5 "一件事"叙事方式（新增——对现有场景的重新包装）

> **政策背景：** 国务院《关于进一步优化政务服务提升行政效能推动"高效办成一件事"的指导意见》（2024年）明确将"一件事一次办"向基层治理延伸。

#### 6.5.1 "消防检查一件事"

**过去：** 检查员到商铺→从包翻纸质表→逐项打勾→有遗漏回去补→回办公室手写报告→交领导审核→存档。一个商铺40分钟，漏项率15%。

**现在：** AI眼镜扫码→HUD逐项引导→AI自动识别隐患（灭火器有效期/通道宽度/电线状态）→发现纸箱堵通道自动拍照标记→检查结束即自动生成结构化报告→报告自动推送消防支队系统。**一个商铺15分钟，漏项率<2%。**

> **一件事闭环：** 扫码签到→AR引导检查→AI隐患识别→自动生成报告→整改通知书一键下发→复查验证→全流程数据沉淀

#### 6.5.2 "矛盾纠纷调解一件事"

**过去：** 两户因房屋边界纠纷→网格员凭经验调解→不成报综治中心→协调多部门→反复沟通→短则3天、长则2周。

**现在：** 扫门牌获取两户基础信息→AI推送相似案例+法规条文+推荐策略→必要时AI眼镜直播、后方专家远程标注指导→GIS比对历史测绘数据→当场形成调解方案→在线签署调解文书→数据沉淀为案例参考。**10-30分钟当场化解，复杂纠纷不超24小时。**

> **一件事闭环：** 扫码获取背景→AI推送调解参考→远程专家支援→在线签署调解文书→数据回流案例库

#### 6.5.3 "独居老人关爱一件事"

**过去：** 老人出意外→邻居发现→打120或联系社区→社区联系家属→耗时数小时至数天。

**现在：** 北斗胸卡检测24小时无外出→系统自动生成"上门查看"任务→匹配最近网格员→赣政通推送→到场扫码确认→敲门发现异常→联系家属送医→结果回传→民政部门实时掌握。**从预警到送医42分钟。**

> **一件事闭环：** 轨迹异常检测→自动生成关爱任务→网格员上门→扫码确认到场→处置结果回传→关爱等级动态调整

#### 6.5.4 "违建治理一件事"

**过去：** 人工巡查发现→上报层层审批→多部门推诿→处置周期长→违建已完工。

**现在：** 无人机自动巡飞→AI识别疑似违建→自动生成工单→匹配城管网格员→AI眼镜到场核查→拍照对比历史影像→确认违建→流转城管/住建→处置后无人机复飞验证→闭环。**发现到处置周期缩短66%。**

> **一件事闭环：** 无人机自动发现→AI识别→工单自动生成→地面核查→多部门协同处置→复飞验证→数据沉淀训练模型
