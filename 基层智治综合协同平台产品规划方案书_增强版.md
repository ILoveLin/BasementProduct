# 一、建设背景与政策依据

## 1.1 现实痛点

- **任务繁重**：基层干部面临"上面千条线，下面一根针"，数据重复采集、多头报表。
- **条块分割**：公安、综治、住建等系统不互通，事件需要人工跨部门协调。
- **数据孤岛**：同一地址、同一人员信息在不同系统反复录入，且无法共享。

## 1.2 政策要求

- 习近平总书记强调："推动社会治理重心向基层下移，运用前沿技术推动城市管理创新。"
- 中央《关于加强基层治理体系和能力现代化建设的意见》要求"构建网格化管理、精细化服务、信息化支撑的基层治理平台"。
- "十五五"规划明确提出"民生诉求统一治理、线上线下协同"。
- 省委省政府、市委市政府部署"为基层减负增效"。

## 1.3 平台定位

本平台不替代任何现有部门业务系统，而是做**"赋能工具"和"协同枢纽"**，通过数据流动与智能协同，为基层干部减负松绑。

# 二、【新增】建设策略对比：为什么选这条路径？

> **补充理由：** 任何信息化项目立项时，决策者必问的三句话——"能不能直接买？""能不能基于现有系统改？""为什么非要从头建？"本章节为方案提供可辩护的路线选择依据。

## 2.1 四种建设路径对比

| 对比维度 | 路径A：采购成熟产品 | 路径B：基于省级平台扩展 | 路径C：多供应商集成 | **路径D：自主构建底座+协同层（本方案）** |
|---------|-------------------|---------------------|-------------------|---------------------------------------|
| **代表方案** | 某厂商网格化平台 / 钉钉政务版 | 省政法委数字治理底座 + 二次开发 | MQ + ESB + 多厂商模块拼装 | 自建1+4+1+N架构 |
| **初期投入** | 低（500万-800万） | 中（800万-1200万） | 中高（1000万-1500万） | 中（1000万-1300万，含4区县试点） |
| **年度运维** | 高（许可费+定制费） | 低-中 | 高（多厂商协调） | 中 |
| **数据自主权** | 低（数据存于厂商） | 低（依赖省级平台迭代节奏） | 中（分散管理） | **高（数据本地化）** |
| **响应本地需求速度** | 慢（依赖厂商排期） | 慢（省级审批流程） | 中 | **快（自有团队）** |
| **跨部门适配能力** | 差（标准产品难适配） | 中 | 中 | **强（自定标准）** |
| **与"E呼即办"协同** | 难（需二次对接） | 难（需协调省级接口） | 可 | **天然一体（同一架构）** |
| **长期演进风险** | 高（锁定厂商） | 高（省级平台替代风险） | 高（集成复杂度指数增长） | **可控（自有技术栈）** |

## 2.2 选型结论

**推荐路径D**，核心理由：

1. **数据安全合规**：一标三实涉及公民敏感信息（人口、房屋、重点人员），必须数据本地化，不可外包存储。
2. **差异化需求密集**：基层治理的"最后一公里"场景极其碎片化（河长巡查、护学岗、独居老人关怀等），标准产品覆盖不了，定制成本远超自建。
3. **与"E呼即办"深度耦合**：本平台与全市E呼即办热线系统为双平台联动，依赖统一的地址编码体系和事件模型，外部产品无法做到无缝对接。
4. **避免供应商锁定**：政府信息化项目生命周期通常10年以上，锁定单一供应商的长期成本为初期投入的3-5倍。

## 2.3 竞品/替代方案参考

| 方案类型 | 代表性案例 | 核心差异与本平台优势 |
|---------|-----------|-------------------|
| 省级统建网格化平台 | 浙江"基层治理四平台"、广东"粤治慧" | 省级平台侧重标准化和统一管理；本平台侧重**本地化灵活适配** + **AI眼镜/大模型等前沿能力注入**，可在省级框架下做差异化创新 |

# 三、总体架构："1+4+1+N"

我们构建了分层解耦、逻辑清晰的架构：

```
┌─────────────────────────────────────────────┐
│            N个多场景（信息采集、检查、        │
│            巡逻、公告、应急协同等）           │
├─────────────────────────────────────────────┤
│         四大共性功能（周边、格友圈、         │
│         二维码门牌、AI智能眼镜）             │
├──────────────────────┬──────────────────────┤
│   AI大模型引擎（智慧大脑）                   │
├──────────────────────┴──────────────────────┤
│   基层治理平台数据底座中台（一标三实+二维码门牌） │
```

- **数据底座中台（核心）**：为四大共性功能提供基础数据和平台支撑。
- **AI智能引擎**：是智慧大脑，向数据底座和四大共性功能注入智能化能力。
- **四大共性功能**：平台的核心功能点和竞争力。
- **N个多场景**：信息采集、检查、巡逻、公告、应急协同等。

# 四、第一个"1"：基层治理平台（数据底座）

## 4.1 核心构成

**二维码门牌**就像每个地址的"健康码"，一扫就知道这个地址下住了谁、有什么单位、有什么风险；**一标三实**就是这些信息的具体台账。

- **二维码门牌**：为每一栋房屋、每一个单位赋予唯一的"数字身份证"。它是物理世界与数字世界的连接入口。
- **一标三实**：标准地址、实有人口、实有房屋、实有单位——底座的核心数据内容。

## 4.2 为什么底座至关重要？

过去，公安、综治、住建各建系统，地址命名不统一（例如同一栋楼在不同系统里地址写法不同），数据无法交换。基层网格员采集的人口信息，公安不能直接用，综治还要再录一遍。

本底座统一了全市地址编码和数据标准，通过一个门牌串联所有条线需要的字段，实现 **"一次采集、多方复用"**，从源头解决数据孤岛。

## 4.3 我们做了什么？

建设了二维码数据底座中台，核心夯实 "一标三实"：
- 为全市每个房屋、每个单位生成唯一的"数字身份证"，并关联二维码门牌。
- 汇聚公安、综治、住建、物业等多源数据，形成"辖区全域块数据"。

## 4.4 带来了什么效果？

| 效果维度 | 具体数据（来自已落地四区县） |
|---------|--------------------------|
| 减少重复填报 | 网格员信息采集时间从15分钟/次 → 3分钟/次 |
| 提升派单精度 | 从"人工微信@人" → 系统自动匹配责任民警/网格员，准确率95%+ |
| 保持数据鲜活 | 通过AI眼镜+二维码门牌动态更新，底数准确率99%以上 |

### 7.3.2 基层信息采集模块

**模块定义：** 以标准地址为锚点，对"人、房、企、物"等治理要素进行新增、修改、删除、查询的全生命周期管理。

**核心动作：** 增、删、改、查（CRUD）

**子功能范围：** 人口采集、房屋采集、通用场所采集、地图标点/图斑采集、从业人员登记、重点人员标注、关注群体标记、出租房管理、标签管理、业户商户上报、自主上报、走访记录、楼栋管理

#### 7.3.2.1 匹配条件（五项必须全部满足）

**条件1 — 用户要素：**

| 允许的角色 | 说明 |
|-----------|------|
| 网格员 | 可采集辖区内所有类型数据 |
| 民警/派出所人员 | 可采集人口、房屋、重点人员 |
| 综治工作人员 | 可采集人口、场所 |
| 物业人员 | 仅可采集本小区房屋和人口 |
| 便民服务员 | 可代为采集群众基本信息 |
| 住建工作人员 | 可采集房屋和场所信息 |

> 无采集权限的角色（如普通群众、纯审批角色）不能进入此模块。

**条件2 — 地址要素（数据必须绑定到一个有效的标准地址）：**

| 方式 | 优先级 | 示例 |
|------|--------|------|
| 扫描标准地址二维码门牌 | 最高 | 扫门上二维码→获取"XX路12号3栋502室" |
| 从任务单带入 | 高 | 系统下发的走访任务自动关联目标地址 |
| 手动选择地址树 | 中 | 省→市→区→街道→社区→门牌号逐级选 |
| 历史记录回填 | 低 | 最近使用过的地址列表 |

> 无标准地址不能进入（可以代码自动生成地址，后续有标准地址直接替换即可）。

**条件3 — 业务意图要素：**

| 操作类型 | 示例 |
|---------|------|
| 新增 | "新增人口""新登记从业人员""标注重点人员" |
| 修改 | "修改联系电话""更新工作单位""变更出租状态" |
| 删除/注销 | "注销人口""解除出租关系" |
| 查询 | 在采集界面中按条件检索已采集的数据 |

**条件4 — 前置状态要素：**

| 操作 | 前置条件 | 校验规则 |
|------|---------|---------|
| 新增 | 目标对象在当前地址下不存在 | 系统自动查重，重复则提示 |
| 修改 | 目标对象已存在且用户有修改权限 | 只能修改自己辖区内的数据 |
| 删除 | 目标对象未被其他业务引用 | 关联未办结事件则不可删除 |
| 批量操作 | 目标列表非空，均在权限范围内 | 逐条校验权限 |

**条件5 — 设备要素和AI眼镜评估：**

| 必需 | 可选增强 | AI眼镜推荐 |
|------|---------|-----------|
| 智能手机 | AI眼镜 | ✅ ⭐⭐⭐ 推荐 |

### 7.3.3 巡查轨迹管理模块

**模块定义：** 记录基层工作人员在物理空间中移动巡查的轨迹、时间、打卡动作及途中发现的问题，形成可追溯的巡查记录。

**核心动作：** 开始巡查 → 移动 → 打卡 → 发现 → 上报 → 结束巡查

**子功能范围：** 巡查路线、巡更打卡、护学打卡、任务巡查、走访日志

#### 7.3.3.1 匹配条件（五项必须全部满足）

**条件1 — 用户要素：**

| 允许的角色 | 典型场景 |
|-----------|---------|
| 网格员 | 日常网格巡查 |
| 民警/辅警 | 治安巡逻、夜间巡逻 |
| 综治巡逻员 | 街面秩序巡查 |
| 护学岗人员 | 上下学校门口巡查 |
| 河长/林长 | 河道/山林巡查 |

**条件2 — 地址要素：**

| 要求 | 说明 |
|------|------|
| 必须现场扫码获取 | 确保"人到现场"，防止虚假打卡 |
| 或地理围栏自动触发 | GPS定位进入预设范围时自动确认 |

**条件3 — 业务意图要素：**

| 意图类型 | 具体操作 |
|---------|---------|
| 开始巡查 | 启动轨迹记录，关联巡查计划 |
| 到达打卡 | 在预设打卡点扫码/地理围栏确认到岗 |

> **这是AI眼镜的杀手级场景！**
> 安全检查的三大痛点 ↔ AI眼镜三大能力完美对应：

| 痛点 | AI眼镜方案 |
|------|-----------|

|--------|---------|

| **离线能力** | 地下室/偏远工厂无网络→端侧AI离线检查→回网后自动同步 |

#### 巡查轨迹(7.3.3)与安全检查(7.3.6)的关键区别

| 维度 | 巡查轨迹(7.3.3) | 安全检查(7.3.6) |
|------|---------------|---------------|
| 核心行为 | 移动中的轨迹记录+路过式观察 | 到达后的定点逐项检查 |

| 时间特征 | 边走边记，持续1-3小时 | 到达后停留15-40分钟 |
| AI眼镜价值 | 免掏手机、自动打卡 | AR引导+AI识别+自动报告（杀手场景） |

# 八、【新增】需求评审与共性判定机制

> **补充理由：** 原方案书第6阶段流程图中，"是否存在共性功能 → 抽象为通用模块"是影响平台架构演进的核心决策节点，但方案书正文未说明由谁、以什么标准做出这个决策。本章节补全这一关键机制。

## 8.1 需求评审委员会组成

| 角色 | 人员 | 职责 |
|------|------|------|
| **产品负责人（主席）** | 1名 | 主持会议，对争议项有最终裁决权，确保评审不偏离平台定位 |
| **业务条线代表** | 每涉及条线1名（公安/综治/住建/城管/民政等） | 说明业务需求合理性，对"是否为本条线共性需求"提供判断 |
| **技术架构师** | 1名 | 评估技术可行性，判断是否具备抽象为通用模块的技术条件 |
| **一线用户代表** | 1-2名网格员/民警 | 从实际使用者角度验证需求的真实性和优先级 |
| **数据底座负责人** | 1名 | 评估新需求对一标三实数据模型的影响 |
| **记录员** | 1名 | 记录评审结论、待办事项，更新N场景模块匹配规则手册 |

## 8.2 评审触发条件

以下任一条件满足时，必须召开需求评审：

1. 五大要素判定后，在十大模块中**未找到匹配**的模块
2. 业务部门提出**跨条线**需求（涉及2个及以上部门）
3. 现有模块**无法满足**需求的核心流程，需大量修改
4. 需求涉及**新硬件设备**（如AI眼镜新外设、传感器等）

## 8.3 共性判定标准（"是否抽象为通用模块"的决策矩阵）

| 判定维度 | 权重 | 得分标准 |
|---------|------|---------|
| **跨条线复用度** | 40% | ≥3个条线有同类需求 → 5分；2个条线 → 3分；仅1个条线 → 1分 |
| **与现有模块的耦合度** | 25% | 与现有模块高度解耦，可独立运行 → 5分；需深度改造现有模块 → 1分 |
| **数据底座依赖度** | 20% | 必须依赖一标三实/二维码门牌 → 5分；与底座无关 → 1分 |
| **预期使用频率** | 15% | 日均≥100次 → 5分；日均<10次 → 1分 |

**决策规则：**
- **综合得分 ≥ 3.5 → 抽象为通用模块**，纳入N场景体系（如新增 6.2.12、6.2.13...）
- **2.0 ≤ 得分 < 3.5 → 暂不抽象，跟踪6个月后再评审**
- **得分 < 2.0 → 个性独立开发**，不纳入N场景模块体系

## 8.4 评审周期

- **常规评审**：每两周一次，集中评审周期内的新需求
- **紧急评审**：涉及重大安全隐患或上级交办紧急任务时，48小时内召集

# 十一、【新增】风险识别与应对策略

> **补充理由：** 任何产品方案书如果不谈风险，就是不完整的。本章节系统性识别6大类风险，并给出具体应对方案。

## 11.1 风险矩阵总览

| 编号 | 风险类别 | 风险描述 | 概率 | 影响 | 等级 |
|------|---------|---------|------|------|------|
| R1 | 数据安全 | 一标三实含公民敏感信息，发生数据泄露 | 中 | 极高 | 高 |
| R2 | 组织阻力 | 部门不愿共享数据，平台成为"空壳" | 高 | 高 | 🔴 高 |
| R3 | 设备供应链 | AI眼镜依赖单一供应商，断供/涨价 | 中 | 高 | 🟡 中高 |
| R4 | 基层抵触 | 网格员认为眼镜是"监控工具"，消极使用 | 中 | 中 | 🟡 中 |
| R5 | 政策变动 | 省级统建平台要求地市停用自建系统 | 低 | 极高 | 🟡 中 |
| R6 | 网络覆盖 | 地下室/农村无信号，离线功能不可靠 | 中 | 中 | 🟢 中低 |
| R7 | AI误判 | 大模型事件分派/隐患识别出错造成后果 | 中 | 中 | 🟡 中 |
| R8 | 技术债务 | 快速迭代导致的架构腐化、性能退化 | 中 | 中 | 🟢 中低 |

## 11.2 各风险详细应对

### R1：数据安全风险

**应对方案：**

- 部署于政务云，与互联网物理隔离
- 数据分级分类：敏感字段（身份证号、重点人员标签）加密存储，访问全程审计
- 采用"最小权限原则"：网格员只能看本辖区，上级领导只能看脱敏汇总
- 每季度第三方安全渗透测试
- 发生数据泄露时的应急预案：2小时内锁定可疑账号、24小时内上报网信办

### R2：部门数据共享阻力

**应对方案：**
- 由政法委牵头签署《数据共享责任书》，将数据共享质量纳入部门年度考核
- 初期不要求部门"交数据"，而是提供"数据对比服务"——让部门先尝到甜头（比如公安通过底座的地址匹配能力发现未掌握的出租房）
- 技术上支持"数据不出域"模式——部门数据的原始库不动，底座通过数据目录+联邦查询的方式按需获取

### R3：AI眼镜供应链风险

**应对方案：**

- 至少认证2家AI眼镜供应商，签订互备协议
- 眼镜端做品牌无关的标准化接入层（功能通过标准API调用，不绑定任何厂家SDK），切换供应商仅需适配硬件层
- 储备关键备件的6个月库存

### R4：基层抵触风险

**应对方案：**
- 推广策略从"强制使用"改为"自愿试点+正向激励"——先让一批种子用户见到效率提升，形成口碑
- 在系统设计中明确：管理者无法查看网格员的实时轨迹，只能看是否完成打卡——减少"被监控感"
- 每月收集一线使用反馈，高频痛点优先优化
- 官方宣传口径始终强调"减负工具"而非"监管工具"

### R5：省级政策变动风险

**应对方案：**
- 本平台的一标三实数据模型采用国标地址编码（GB/T 35639-2017），确保与省级平台数据可互通
- 接口层采用标准RESTful API，可对接任何省级平台的开放接口
- 核心模块（AI眼镜+安全巡检+二维码门牌）为差异化创新，省级平台短期难以覆盖
- 最坏情况：平台降级为省级底座之上的"应用层模块"，核心技术资产不损失

### R6：网络覆盖不足

**应对方案：**
- 所有核心操作（采集、巡查打卡、检查判定）设计为离线优先，数据本地缓存，回网后增量同步
- 关键场景（地下室检查、偏远工厂）预设离线检查清单，端侧AI完成隐患识别
- 离线数据同步采用断点续传+冲突检测机制

### R7：AI误判风险

**应对方案：**
- AI分派/识别结果始终以"建议"而非"决定"形态呈现——人工确认后才生效
- 建立错误反馈闭环：每次人工修正AI建议时，自动记录为训练样本
- 对高风险场景（如重点人员标注、安全隐患判定）设置人工复核节点
- 定期输出AI准确率报告，准确率低于阈值自动降级为纯规则引擎

### R8：技术债务

**应对方案：**

- 每季度一次架构Review，识别性能瓶颈和代码腐化区域
- 核心模块单元测试覆盖率要求≥70%
- 每年度安排"技术健康"专项预算（占总预算的15%-20%），用于重构、升级依赖、性能优化
